라이브 dashboard 의 통계적 한계 — 솔직 명시

메인 보고서의 Failures Wall 은 backtest 단계에서 버린 가설들이고, 여기는 지금 라이브 paper trade dashboard 가 절대 평가에 못 쓰이는 이유다.

01. 표본 너무 작음 — closed n ≈ 28 / 24, 일자별 aggregation 시 unique day n 가 한 자리 수. Sharpe / Sortino 의 standard error 가 매우 큼. bootstrap 95% CI 가 0 을 가로질러 통계적 유의 X. 30+ 거래가 쌓일 때까지 결과는 *지표 모양만* 으로 본다.
02. survivorship bias 미처리 — pyupbit 의 현재 listing 만 가져옴. 과거 상폐 코인이 panel 에서 빠져 historical EV 가 부풀 수 있음. Upbit 공시 archive 로 historical universe 재구성 필요.
03. 거래비용 0.15% 가정의 낙관성 — ≥20% tail pump 코인은 호가 sparse → 실제 진입 시 시장 충격 + 슬리피지 0.5~1.5% 가 일반. cost 모델을 ATR / volume 기반으로 강화할 필요.
04. regime-dependent — BTC bull regime 에서만 동작 설계. bear 구간은 침묵. 전체 시장 cycle 을 cover 하지 못하므로 누적 결과는 시기 의존성 큼.
05. 5% TP 룰의 단순성 — sweep 결과 대신 사용자 의사결정에 맞춘 룰. avg_max +6.82% 가 5% TP 에서 잘려 +20% tail 을 놓침. 룰 민감도 분석 (3% / 7% / no TP) 추후 추가.
