이번 주 이야기

우편번호 "06152"와 숫자 6152는 어떻게 다를까요?

"06152"에서 앞의 0을 지우면 6152가 됩니다. 하지만 우편번호에서 앞의 0은 의미가 있습니다 — 없애면 다른 우편번호가 됩니다.
숫자 6152에서 앞에 0을 붙이면? 그냥 6152입니다. 숫자에서 앞의 0은 의미가 없습니다.

같아 보이는 "숫자들"이 왜 다르게 동작할까요?
그 이유가 바로 타입(Type)입니다.
이번 주 핵심 질문

컴퓨터의 메모리에는 오직 0과 1의 나열만 저장됩니다. 그렇다면 65라는 숫자와 'A'라는 문자는 어떻게 구분될까요?

실제로 비트로 보면: 01000001
이 비트열을 정수(int)로 읽으면 → 65
이 비트열을 문자(str)로 읽으면 → 'A' (ASCII)

타입은 "이 비트들이 무엇을 의미하는가"를 결정하는 규칙입니다.

왜 타입이 필요한가?

초급

같은 모양, 다른 의미

"3"과 3은 같을까요, 다를까요?
화면에 보이는 모습은 비슷하지만 컴퓨터 안에서는 완전히 다른 것입니다.

문자열 "3"과 숫자 3의 차이 
print("3" + "3")   # 문자열 이어붙이기 → "33"
print(3 + 3)       # 숫자 더하기 → 6
실행 결과:

같은 + 연산자를 사용했는데 결과가 다릅니다.
컴퓨터는 타입을 보고 "이 +가 더하기인지, 이어붙이기인지"를 결정합니다.

핵심 비유: 타입은 박스의 레이블입니다.
같은 크기의 박스라도 "음식" 레이블이 붙은 것과 "폭발물" 레이블이 붙은 것은 다르게 취급됩니다.
컴퓨터도 비트를 보기 전에 먼저 타입(레이블)을 봅니다.

비트와 해석 — 타입이 의미를 만든다

컴퓨터 메모리는 오직 0과 1의 나열만 저장합니다. type() 함수로 파이썬이 어떻게 타입을 구분하는지 확인해봅시다.

type()으로 타입 확인하기

Python Data Model — 모든 것은 객체

WHY — Python 공식 문서가 말하는 타입

Python 3.14 Data Model(데이터 모델) 문서는 이렇게 시작합니다:
"Every object has an identity, a type and a value."
(모든 객체는 아이덴티티, 타입, 값을 갖습니다.)

  • 아이덴티티(identity): 메모리 주소 — id(x)로 확인
  • 타입(type): 어떤 연산을 지원하는지 정의 — type(x)로 확인
  • 값(value): 실제 데이터 — print(x)로 확인
객체의 3요소 확인하기 
x = 42
print(id(x))      # 아이덴티티: 메모리 주소 (예: 140234866532640)
print(type(x))    # 타입: <class 'int'>
print(x)          # 값: 42

# 타입은 가능한 연산을 결정한다
print(42 + 1)     # int: 덧셈 가능
# print("A" + 1)  # TypeError: str + int 불가 — 타입이 다름

타입이 연산을 결정한다는 것이 핵심입니다. "A" + 1이 불가능한 이유는 "문자와 숫자를 더하는 방법"이 정의되지 않았기 때문입니다. 타입 시스템은 이런 무의미한 연산을 막아주는 안전망입니다.

CPython PyObject 구조 + 타이핑 이론

Part A — CPython: 모든 것은 PyObject*

CPython에서 파이썬의 모든 값은 C 구조체 PyObject로 표현됩니다. cpython/Include/object.h에 정의되어 있습니다.

CPython PyObject 구조 (C 코드) 
/* CPython Include/object.h (단순화) */
typedef struct _object {
    Py_ssize_t  ob_refcnt;   /* 참조 카운트: 이 객체를 가리키는 포인터 수 */
    PyTypeObject *ob_type;   /* 타입 포인터: 이 객체의 타입 객체를 가리킴 */
} PyObject;

/* 예: int 42는 실제로 이렇게 저장됨 */
typedef struct {
    PyObject   ob_base;     /* ob_refcnt + ob_type */
    Py_ssize_t ob_size;     /* 자릿수 */
    digit      ob_digit[1]; /* 실제 값 (가변 길이 배열) */
} PyLongObject;

ob_type 포인터가 핵심입니다. 42라는 객체의 ob_type은 &PyLong_Type을 가리킵니다. 파이썬이 42 + 1을 계산할 때, 실제로는 ob_type->tp_as_number->nb_add(42, 1)을 호출합니다.

Python에서 ob_type 확인하기 
import sys
x = 42
print(type(x))            # <class 'int'>  ← ob_type이 가리키는 타입 객체
print(sys.getsizeof(x))   # 28 bytes (작은 int)
print(x.__class__)        # <class 'int'>  ← ob_type 접근

# 타입 객체도 객체다
print(type(int))          # <class 'type'>  ← int의 타입은 type(메타클래스)
Part B — 타이핑 이론: 동적 vs 정적 vs 덕 타이핑

이 CS 이론이 필요한 이유: Python이 왜 이런 타입 시스템을 가졌는지 설명하려면 타이핑 철학을 이해해야 합니다.

세 가지 타이핑 방식 비교 
# 1. 정적 타이핑 (Static Typing) — 예: Java, C
# int x = 42;  ← 컴파일 시점에 타입 결정, 변경 불가
# x = "hello"; ← 컴파일 에러

# 2. 동적 타이핑 (Dynamic Typing) — Python
x = 42          # 런타임에 타입 결정
x = "hello"     # OK — 같은 변수가 다른 타입을 가질 수 있음
x = [1, 2, 3]   # OK

# 3. 덕 타이핑 (Duck Typing) — Python의 핵심
# "오리처럼 걷고 오리처럼 꽥꽥거리면 오리다"
# 타입 이름이 아니라 "무엇을 할 수 있는가"로 판단
def add(a, b):
    return a + b   # int, float, str 모두 + 를 지원하면 동작함

print(add(1, 2))       # 3
print(add(1.0, 2.0))   # 3.0
print(add("a", "b"))   # "ab"

CPython의 타입 시스템은 동적 타이핑 + 덕 타이핑의 조합입니다. 변수의 타입은 런타임에 결정되고(ob_type 포인터로 관리), 연산 가능 여부는 해당 타입 객체에 해당 슬롯(tp_as_number, tp_as_sequence 등)이 정의되어 있는지로 판단합니다.

int — 정수

WHY — 왜 int가 존재하는가?

개수를 세거나, 순서를 나타내거나, 인덱스를 다룰 때 소수점이 필요 없습니다. 3.0번째 항목은 의미가 없습니다. 3번째 항목만 의미가 있습니다.
int는 "소수점 없이 정확한 정수 연산"을 위해 존재합니다.

기본 정수 연산 
print(2 + 3)    # 덧셈: 5           (사과 2개 + 3개)
print(10 - 4)   # 뺄셈: 6           (10명 중 4명 퇴장)

Python int의 특별함 — 크기 제한이 없다

WHY — 파이썬 int는 왜 overflow가 없는가?

C나 Java의 int는 32비트 또는 64비트로 고정되어, 일정 크기를 넘으면 overflow(넘침)가 발생합니다.
파이썬의 int는 임의 정밀도(arbitrary precision)를 지원합니다. 메모리가 허용하는 한 얼마든지 큰 숫자를 다룰 수 있습니다.

파이썬 int의 임의 정밀도 
# 파이썬은 아무리 큰 정수도 정확하게 계산한다
print(2 ** 100)
# 1267650600228229401496703205376

print(2 ** 1000)  # 301자리 수도 정확히!

# 다른 진수 표현
print(0b1010)   # 2진수: 10
print(0o17)     # 8진수: 15
print(0xff)     # 16진수: 255

# 진수 변환
print(bin(255))  # '0b11111111'
print(oct(255))  # '0o377'
print(hex(255))  # '0xff'
PEP 238 — 왜 /는 항상 float을 돌려주는가?
Python 2에서는 5 / 2가 2(정수)였습니다. 이로 인해 수많은 버그가 발생했습니다 — 프로그래머가 "5를 2로 나누면 2.5"라고 기대하는데 파이썬이 2를 돌려줬기 때문입니다.
PEP 238(2001)에서 제안되어 Python 2.2에서 from __future__ import division으로 먼저 도입되었고, Python 3에서 /는 기본적으로 항상 float을 돌려주는 것이 표준이 되었습니다. "명시적인 것이 암묵적인 것보다 낫다(Explicit is better than implicit)" — Zen of Python
🚨 흔한 실수 — is와 == 혼동:
a = 256
b = 256
print(a is b)    # True  ← CPython이 -5~256을 캐싱!

a = 257
b = 257
print(a is b)    # False ← 257은 캐시 범위 밖, 다른 객체

# 규칙: 값 비교는 반드시 ==, 동일 객체 확인에만 is
print(a == b)    # True  ← 값이 같음 (항상 신뢰 가능)
print(a is b)    # False ← 같은 객체가 아님
결론: 숫자·문자열 비교는 항상 ==. is는 is None, is not None에서만 사용하세요.

int 연산의 세부 동작 + 정수 나눗셈

정수 나눗셈의 정확한 동작 
# // 연산: 수학적 바닥 나눗셈(floor division)
# 결과는 항상 "아래 방향으로" 내림
print(7 // 2)     #  3  (양수: 3.5의 바닥)
print(-7 // 2)    # -4  (음수: -3.5의 바닥 = -4)

# % 연산: 수학적 모듈로
# 결과의 부호는 항상 제수(오른쪽 수)와 같다
print(7 % 2)      #  1
print(-7 % 2)     #  1  (양수 — 제수가 양수이므로)
print(7 % -2)     # -1  (음수 — 제수가 음수이므로)

# 검증: a == (a // b) * b + (a % b)
a, b = -7, 2
print((a // b) * b + (a % b))  # -7
C의 정수 나눗셈과 다릅니다. C에서 -7 / 2는 -3(0 방향 절단), 파이썬에서는 -4(바닥). 파이썬의 방식이 수학적으로 더 일관성이 있습니다.

CPython PyLongObject 구조 + 소정수 캐싱

Part A — CPython: PyLongObject

Part A — CPython: PyLongObject

CPython 3.12+에서 int는 longobject.c에 구현됩니다. 임의 정밀도를 위해 내부적으로 자릿수를 배열로 저장합니다.

PyLongObject 구조 (단순화) 
/* CPython 3.11 이하 — Include/cpython/longintrepr.h (단순화) */
struct PyLongObject {
    PyObject_VAR_HEAD      /* ob_refcnt, ob_type, ob_size */
    digit ob_digit[1];     /* 자릿수 배열 (기수 2^30) */
};

/* Python 3.12+: 내부 구조가 재설계됨 (compact representation, _PyLongValue union)
   그러나 기수 2^30, 임의 정밀도 원리는 동일하게 유지됨 */
/* 예: 1,073,741,824 (= 2^30) 이상의 수는 두 개의 digit이 필요 */
Python에서 확인: 크기와 소정수 캐싱 
import sys

# 정수 크기 — 숫자가 클수록 더 많은 메모리
print(sys.getsizeof(0))          # 28 bytes (Python 3.12+)
print(sys.getsizeof(1))          # 28 bytes
print(sys.getsizeof(10**30))     # 40 bytes (더 큰 배열 필요)
print(sys.getsizeof(10**100))    # 72 bytes

# 소정수 캐싱: -5 ~ 256은 미리 만들어 놓는다
a = 100
b = 100
print(a is b)    # True  — 같은 객체 (캐시된 정수)

# 주의: REPL에서 한 줄씩 입력할 때만 False
# 스크립트로 실행하면 상수 폴딩(constant folding)으로 True가 될 수 있음
# → 정수 비교는 반드시 == 을 사용해야 한다 (is 아님)
a = 1000
b = 1000
# REPL: a is b → False (줄마다 별도 컴파일)
# script: a is b → True (같은 블록 내 리터럴은 폴딩)

Part B — 2의 보수(Two's Complement)

Part B — 2의 보수(Two's Complement)

C int가 왜 overflow하는지 이해하려면 2의 보수가 필요합니다.
32비트 int의 최대값: 2³¹ - 1 = 2,147,483,647
여기에 1을 더하면: -2,147,483,648 (wrap-around)

Python int는 ob_digit[] 배열을 동적으로 늘리기 때문에 이런 제한이 없습니다. 이것이 Python의 "임의 정밀도 정수"가 의미하는 것입니다.

float — 실수와 IEEE 754

WHY — 왜 float이 int와 분리되어 있는가?

3.14, 1.6, -0.5 같은 소수는 정수와 다르게 표현해야 합니다. 컴퓨터는 소수를 부동소수점(floating point) 방식으로 저장합니다. 이 방식은 매우 크거나 작은 수를 표현할 수 있지만, 모든 소수를 정확히 표현하지는 못합니다.

왜 0.1 + 0.2 ≠ 0.3인가?

WHY — 부동소수점 오차의 원인

0.1을 이진수로 변환하면 무한소수가 됩니다.
십진수에서 1/3 = 0.3333...이 무한소수인 것처럼, 이진수에서 0.1은 0.0001100110011...로 무한히 반복됩니다.
컴퓨터는 이것을 64비트에 맞게 잘라서 저장하므로, 미세한 오차가 생깁니다.

# 해결책 1: round()로 반올림

print(round(0.1 + 0.2, 1) == 0.3)  # True

# 해결책 2: math.isclose() (권장)

print(math.isclose(0.1 + 0.2, 0.3))  # True

# 0.1의 실제 저장값 확인

# 0.1000000000000000055511151231257827021181583404541015625

IEEE 754 — 부동소수점의 국제 표준

Python의 float는 IEEE 754 배정밀도(double precision) 64비트 형식을 사용합니다. 이것은 C의 double과 동일합니다.

IEEE 754 64비트 구조 
# IEEE 754 double (64비트) 구조:
# [1비트 부호] [11비트 지수] [52비트 가수]

# struct 모듈로 실제 비트 확인
import struct

def float_to_bits(f):
    # '>d' = 빅엔디언 double → IEEE 754 표준 비트 순서로 표시
    # 리틀엔디언(x86) 시스템에서 'd'만 쓰면 바이트 순서가 뒤집힌다
    return struct.pack('>d', f).hex()

print(float_to_bits(1.0))   # 3ff0000000000000
print(float_to_bits(0.1))   # 3fb999999999999a
print(float_to_bits(-1.0))  # bff0000000000000
print(float_to_bits(float('inf')))  # 7ff0000000000000 (무한대)

0.1의 비트 패턴(3fb999999...)을 보면 9999...가 반복되는 것이 보입니다 — 이진 무한소수를 잘라낸 흔적입니다.

print(float('nan'))   # Not a Number

# NaN의 특이한 성질
nan = float('nan')
print(nan == nan)     # False (NaN은 자기 자신과도 같지 않다!)
print(math.isnan(nan))  # True (NaN 검사는 isnan() 사용)

print(1 / 0)  # ZeroDivisionError (파이썬은 C와 달리 예외 발생)

Part B — Unicode 표준: Code Point vs Grapheme Cluster

Part B — Unicode 표준: Code Point vs Grapheme Cluster

Code Point: Unicode에서 각 문자에 할당된 정수 값 (U+0041 = 'A')
Grapheme Cluster: 사용자가 인식하는 하나의 "글자" 단위

예: "é"는 e(U+0065) + ◌́(U+0301, 결합 악센트)로 표현 가능 (2 code point = 1 grapheme cluster). len("é") == 2일 수 있습니다.

이것이 텍스트 처리를 복잡하게 만드는 이유입니다. Unicode 정규화(NFC, NFD)로 이를 표준화할 수 있습니다.

Unicode 정규화 
import unicodedata

# 두 가지 방법으로 'é' 표현
e1 = "\u00e9"          # NFC: é (단일 code point)
e2 = "e\u0301"         # NFD: e + 결합 악센트 (2 code points)

print(e1 == e2)          # False (다른 코드 포인트 시퀀스)
print(len(e1), len(e2))  # 1  2

# 정규화로 통일
n1 = unicodedata.normalize("NFC", e1)
n2 = unicodedata.normalize("NFC", e2)
print(n1 == n2)          # True

WHY — 왜 bool은 int의 서브클래스인가?

# PEP 285: bool은 반드시 True 또는 False 중 하나
# bool을 상속하더라도 새 인스턴스를 만들 수 없다
# bool(x)는 x.__bool__() 또는 x.__len__()을 호출한다

class MyList:
    def __init__(self, items):
        self.items = items
    def __bool__(self):
        return len(self.items) > 0   # 비어있으면 False

ml_empty = MyList([])
ml_full  = MyList([1, 2, 3])
print(bool(ml_empty))   # False
print(bool(ml_full))    # True
if ml_full:
    print("항목이 있습니다")  # 출력됨

파이썬이 if x:를 평가할 때 내부적으로 x.__bool__()을 호출합니다. __bool__이 없으면 x.__len__()을 시도하고, 둘 다 없으면 True를 반환합니다. 이것이 덕 타이핑의 예입니다 — bool이 아니어도 bool처럼 동작할 수 있습니다.

WHY — 파이썬은 왜 명시적 변환을 요구하는가?

Zen of Python: "명시적인 것이 암묵적인 것보다 낫다."
자동 변환은 코드의 의도를 불명확하게 만들고, 숨어있는 버그의 원인이 됩니다.

# int(x)는 내부적으로 x.__int__()를 호출한다
class Temperature:

        return float(self.celsius)

print(float(t))  # 36.6
print(str(t))    # 36.6°C

LLM의 산술 약점과 타입 추론 시스템

Part A — LLM의 산술 약점: BPE와 토큰 단위 처리

왜 LLM은 수학 계산에 약한가?

GPT/Claude 같은 LLM은 BPE(Byte Pair Encoding)로 텍스트를 토큰으로 분해합니다.
숫자 1234는 하나의 토큰이 될 수 있지만, 큰 숫자의 덧셈은 토큰 단위로 처리되지 않습니다.

LLM은 다음 토큰의 확률 분포를 학습한 것이지, 실제 산술 회로가 없습니다. 99999 + 1을 틀리는 이유가 여기 있습니다.

이것이 LLM이 생성한 코드에서 타입 변환 오류가 많은 이유 중 하나입니다: LLM은 "다음에 올 것 같은 토큰"을 예측하지, 코드를 실제로 실행해서 검증하지 않습니다.

문제 8 전공자

sys.getsizeof(0)은 28이고, sys.getsizeof(10**100)은 72입니다. 이 차이가 나는 이유를 CPython PyLongObject 구조를 이용해 설명하세요.

▶ 정답 보기
확인 코드
import sys
for n in [0, 1, 2**30, 10**30, 10**60, 10**100]:
    print(f"sys.getsizeof({n:.2e}) = {sys.getsizeof(n)} bytes")

설명:
CPython의 PyLongObject는 ob_digit[] 배열에 숫자를 저장합니다. 각 digit은 기수 2³⁰(약 10억) 단위입니다.

0: ob_digit 배열 크기 0이지만 최소 1 슬롯 할당 = 28 bytes
1: ob_digit[0] = 1 → 1 digit = 28 bytes
10**100: 100자리 십진수는 약 11개의 2³⁰ 단위 필요 → 72 bytes

숫자가 커질수록 ob_digit 배열이 늘어나므로 메모리도 늘어납니다. 이것이 Python int가 임의 정밀도를 구현하는 방식입니다.

문제 9 전공자

struct.pack('>d', 0.1).hex()의 결과를 실행해보고, 이 16진수가 IEEE 754 64비트 구조에서 어떤 의미인지 설명하세요. (부호 비트, 지수 필드, 가수 필드)

▶ 정답 보기
분석 코드
import struct

# '>Q'/'>d' = 빅엔디언 → [부호][지수][가수] 순서 그대로 읽힘
bits = struct.unpack('>Q', struct.pack('>d', 0.1))[0]
print(f"Hex: {struct.pack('>d', 0.1).hex()}")
# 3fb999999999999a  ← [0|01111111011|1001100110011...] 순서

sign    = (bits >> 63) & 1
exp_raw = (bits >> 52) & 0x7FF
frac    = bits & ((1 << 52) - 1)

print(f"부호:  {sign}")              # 0 (양수)
print(f"지수:  {exp_raw} (biased)")  # 1019
print(f"실제 지수: {exp_raw - 1023}") # -4
print(f"가수:  {frac:#014x}")         # 0x999999999999a
# 값 = (-1)^0 × 2^(-4) × 1.b999...₁₆ ≈ 0.1

0.1의 IEEE 754 표현:
- 부호 비트: 0 (양수)
- 지수: 1019 - 1023 = -4 (바이어스 1023을 빼야 실제 지수)
- 가수: 1.999...a × 2^-4 (이진 반복소수로 인해 999... 패턴)

0.1이 이진수로 정확히 표현되지 않는 이유: 1/10을 이진수로 변환하면 0.0001100110011...로 무한히 반복됩니다. 이 반복이 가수 52비트에서 잘려 오차가 생깁니다.

용어 사전

Type (타입)
값이 무엇인지, 어떤 연산을 지원하는지를 정의하는 레이블. 비트에 의미를 부여한다.
int
정수 타입. Python에서는 임의 정밀도 — 크기 제한 없음. CPython: PyLongObject.
float
실수 타입. IEEE 754 64비트 배정밀도. 모든 소수를 정확히 표현하지 못함.
str
문자열 타입. Unicode 기반, 불변(Immutable). Python 3의 기본 텍스트 타입.
bool
논리값 타입. True와 False만 존재. int의 서브클래스. True=1, False=0.
Immutable (불변)
한 번 생성되면 값을 변경할 수 없는 타입. int, float, str, bool, tuple이 해당.
Mutable (가변)
생성 후 내용을 변경할 수 있는 타입. list, dict, set이 해당.
Truthiness
모든 Python 객체가 가지는 True/False 해석. if문에서 bool이 아닌 값을 사용 가능하게 함.
Falsy
bool로 변환 시 False가 되는 값들: 0, 0.0, "", [], {}, set(), (), None, False.
Type Conversion (타입 변환)
값을 한 타입에서 다른 타입으로 변환하는 것. int(), float(), str(), bool() 등.
Explicit Conversion (명시적 변환)
프로그래머가 직접 변환 함수를 호출하는 것. Python이 선호하는 방식.
Implicit Conversion (암묵적 변환)
언어가 자동으로 타입을 변환하는 것. Python은 int→float 같은 제한적 경우만 허용.
IEEE 754
부동소수점 수의 컴퓨터 표현 국제 표준. Python float는 64비트 double precision 사용.
Floating Point (부동소수점)
소수를 [부호][지수][가수] 형태로 저장하는 방식. 큰 범위를 표현하지만 정밀도에 한계.
Arbitrary Precision (임의 정밀도)
크기 제한 없이 정수를 표현하는 방식. Python int의 특성. 메모리가 허용하는 한 무한히 큰 수.
PyObject
CPython에서 모든 Python 값의 기반 C 구조체. ob_refcnt(참조 카운트)와 ob_type(타입 포인터)를 포함.
ob_type
PyObject의 필드. 해당 객체의 타입 객체(PyTypeObject)를 가리키는 포인터. 가능한 연산을 결정.
ob_refcnt
PyObject의 필드. 참조 카운트 — 이 객체를 가리키는 포인터 수. 0이 되면 메모리 해제.
Singleton (싱글턴)
프로그램 전체에서 단 하나의 인스턴스만 존재하는 객체. None, True, False가 해당.
Duck Typing (덕 타이핑)
"오리처럼 걷고 꽥꽥거리면 오리다" — 타입 이름 대신 "무엇을 할 수 있는가"로 판단.
Dynamic Typing (동적 타이핑)
변수의 타입이 런타임에 결정되는 방식. Python은 동적 타이핑. Java/C는 정적 타이핑.
Type Annotation (타입 힌트)
변수나 함수에 예상 타입을 명시하는 문법 (def f(x: int) -> str). 실행 시 강제되지 않음.
Unicode
세상의 모든 문자를 표현하는 국제 표준. Python 3의 str은 Unicode 기반.
Code Point
Unicode에서 각 문자에 할당된 정수 값. 예: 'A' = U+0041 = 65.
Metaclass (메타클래스)
클래스의 클래스. type은 Python의 기본 메타클래스 — 모든 클래스 객체는 type의 인스턴스.
Truncation (절삭)
소수점 이하를 0 방향으로 잘라내는 것. int(3.9)→3, int(-3.9)→-3. math.floor와 다름
Concatenation (이어붙이기)
문자열을 +로 연결하는 연산. "안녕" + "하세요" → "안녕하세요"
PEP 285
bool 타입을 파이썬에 추가한 제안서. bool은 int의 서브클래스로 설계됨
Subclass (서브클래스)
다른 클래스를 상속받아 확장한 클래스. bool은 int의 서브클래스이므로 isinstance(True, int)가 True
Operator Overloading (연산자 오버로딩)
같은 연산자(+)가 타입에 따라 다르게 동작하는 것. 3+2→5(덧셈), "안녕"+"하세요"→"안녕하세요"(이어붙이기)
