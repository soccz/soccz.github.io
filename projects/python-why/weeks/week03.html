이게 가능한 이유는 하나입니다: x와 y는 같은 객체를 가리키고 있었다.

이번 주 핵심 질문

x = 1은 "상자(x)에 값(1)을 넣는다"는 뜻일까요?

아닙니다. Python의 공식 문서(tutorial/classes)는 이렇게 말합니다:
"대입은 데이터를 복사하지 않습니다 — 이름을 단지 객체에 연결할 뿐입니다."

x = 1은 "이름표 x를 객체 1에 붙인다"는 뜻입니다.
이 차이가 위 수수께끼의 해답입니다.

왜 "이름표"인가? — 바인딩 모델

초급

포스트잇 비유

변수를 "상자"로 배운 적 있나요? 값을 상자 안에 넣는다고요.
Python에서는 그렇지 않습니다. 변수는 포스트잇(이름표)입니다.

상자 모델 vs 이름표 모델:

📦 상자 모델: "x라는 상자 안에 1을 넣는다"
    → 상자를 다른 상자에 복사하면 두 개의 독립된 값이 생긴다

🏷️ 이름표 모델: "이름표 x를 객체 1에 붙인다"
    → 이름표를 복사하면 두 이름표가 같은 것을 가리킨다
이름표 모델 확인 
나이 = 10       # 이름표 '나이'를 숫자 10에 붙인다
친구_나이 = 나이  # 이름표 '친구_나이'도 숫자 10에 붙인다
나이 = 20       # 이름표 '나이'를 숫자 20으로 옮긴다

print(나이)      # 20  ← 나이는 이제 20을 가리킨다
print(친구_나이)  # 10  ← 친구_나이는 여전히 10을 가리킨다
실행 결과:
20 10

친구_나이 = 나이는 "10이라는 물건에 친구_나이라는 이름표도 붙인다"는 뜻입니다.
이후 나이 = 20으로 나이 이름표만 이동해도, 친구_나이 이름표는 여전히 10을 가리킵니다.

🚨 함정 — 친구_나이도 바뀔 것 같은데?
나이 = 10
친구_나이 = 나이    # 같은 10을 가리킴
나이 = 20          # 나이 이름표를 20으로 옮김

print(친구_나이)    # 10 ← 20이 아니다!
# 왜? "친구_나이"는 여전히 원래의 10을 가리키고 있음
# 이름표를 옮긴 것이지, 10이라는 값이 20으로 바뀐 것이 아님
✏️ 지금 바로 해봐: 이름표가 바뀌는 것을 직접 확인해보세요.
x = 10
print(id(x))   # 메모리 주소 (숫자)

x = 20         # x가 20을 가리키도록 이름표를 옮김
print(id(x))   # 다른 주소! x가 다른 곳을 가리킴

y = x          # y도 같은 20을 가리킴
print(id(x) == id(y))   # True — 같은 객체!
id()는 객체의 메모리 주소(집 번호)를 알려줍니다. x를 바꿔도 y는 그대로입니다!
중급

바인딩(Binding) — 이름과 객체의 연결

Python에서 x = 1은 세 단계로 일어납니다:

  1. 정수 객체 1이 메모리에 생성(또는 캐시에서 조회)됩니다
  2. 이름 "x"가 그 객체를 가리키도록(참조하도록) 설정됩니다
  3. 네임스페이스(현재 스코프의 딕셔너리)에 {"x": <int 객체>}가 등록됩니다
바인딩의 세 가지 방식 
# 1. 대입(Assignment) — 가장 기본적인 바인딩
x = 42

# 2. 함수 정의(def) — 함수 이름도 바인딩
def greet():
    print("안녕!")
# "greet"라는 이름이 함수 객체에 바인딩됨

# 3. import — 모듈 이름도 바인딩
import math
# "math"라는 이름이 모듈 객체에 바인딩됨

# 바인딩 결과 확인
print(type(greet))   # <class 'function'>
print(type(math))    # <class 'module'>
실행 결과:
<class 'function'> <class 'module'>
Python 공식 문서(tutorial/classes):
"새 이름을 소개하는 모든 연산은 지역 스코프를 사용합니다: 특히, import 문과 함수 정의는 모듈이나 함수 이름을 지역 스코프에 연결합니다."
고급

객체의 세 요소 + 참조 카운트

2주차에서 배운 "모든 객체는 identity, type, value를 갖는다"를 기억하시나요?
바인딩은 이름 → identity(메모리 주소)를 연결하는 것입니다.

바인딩과 객체 추적 
x = [1, 2, 3]
print(f"x의 id: {id(x)}")   # 예: 140234866532640

y = x
print(f"y의 id: {id(y)}")   # 동일한 주소! — 같은 객체

print(x is y)                # True — 이름만 다를 뿐 동일한 객체

# 재바인딩(rebinding): x가 새 객체를 가리키게 된다
x = [4, 5, 6]
print(f"x의 id: {id(x)}")   # 다른 주소 — 새 객체
print(f"y의 id: {id(y)}")   # 여전히 원래 주소 — [1,2,3]

print(x is y)                # False — 이제 다른 객체
실행 결과 (id는 실행마다 다를 수 있음):
x의 id: 140234866532640 y의 id: 140234866532640 True x의 id: 140234866533760 y의 id: 140234866532640 False

x = [4, 5, 6]은 기존 리스트를 수정하는 것이 아니라, 새 리스트 객체를 만들고 x 이름표를 거기로 재바인딩(rebinding)합니다.
y는 그대로 원래 [1, 2, 3] 객체를 가리킵니다.

참조 카운트(Reference Count):
CPython은 각 객체가 몇 개의 이름표(참조)를 받고 있는지 세고 있습니다.
y = x 후 [1,2,3] 객체의 참조 카운트는 2가 됩니다.
x = [4,5,6] 후 [1,2,3] 객체의 참조 카운트는 1로 줄어듭니다.
참조 카운트가 0이 되면 메모리에서 자동 해제됩니다 (GC).
전공자

Part A: STORE_NAME 바이트코드 — 바인딩의 CPython 구현

x = 42가 실행될 때 CPython 인터프리터 내부에서 정확히 무슨 일이 일어나는지 dis 모듈로 확인합니다.

모듈 레벨 대입의 바이트코드 
import dis

code = compile("x = 42", "<string>", "exec")
dis.dis(code)
실행 결과:
0 0 RESUME 0 1 2 LOAD_CONST 0 (42) 4 STORE_NAME 0 (x) 6 RETURN_CONST 1 (None)

두 개의 핵심 명령어:

  • LOAD_CONST 42: 상수 42(PyObject*)를 평가 스택에 push
  • STORE_NAME "x": 스택에서 pop한 PyObject*를 현재 네임스페이스 딕셔너리(f_globals 또는 f_locals)에 {"x": PyObject*}로 저장
함수 내부 대입 — STORE_FAST 
import dis

def foo():
    x = 42

dis.dis(foo)
실행 결과:
0 0 RESUME 0 1 2 LOAD_CONST 1 (42) 4 STORE_FAST 0 (x) 6 RETURN_CONST 0 (None)
STORE_NAME vs STORE_FAST

STORE_NAME — 모듈/클래스 스코프: PyDict_SetItem(locals, name, value)
    딕셔너리 해시 룩업 → 상대적으로 느림

STORE_FAST — 함수 스코프: fastlocals[var_index] = value
    C 배열 인덱스 접근 → 빠름 (컴파일 타임에 인덱스 결정)

함수 내 지역 변수가 빠른 이유: 딕셔너리 룩업 없이 C 배열 직접 접근.
네임스페이스가 딕셔너리임을 직접 확인 
x = 100
y = "hello"

# 모듈 전역 네임스페이스는 실제 딕셔너리
ns = globals()
print(type(ns))          # <class 'dict'>
print(ns["x"])           # 100
print(ns["y"])           # hello

# 직접 네임스페이스를 조작할 수도 있다 (권장하지 않음)
ns["z"] = 999
print(z)                 # 999 — globals()로 추가한 변수가 접근 가능
실행 결과:
<class 'dict'> 100 hello 999

Part B: 값 의미론 vs 참조 의미론

이름표 모델이 "생소하게" 느껴지는 이유: 대부분의 언어가 값 의미론(value semantics)을 기본으로 사용하기 때문입니다.

값 의미론(Value Semantics) — C, Go, Rust (기본)
int x = 5; int y = x;
y는 x의 복사본. x를 바꿔도 y는 영향 없음.
변수는 값을 "직접 보유"한다.

참조 의미론(Reference Semantics) — Python, Java 객체, JavaScript 객체
x = [1,2,3]; y = x
y는 x가 가리키는 같은 객체를 가리킴. 객체를 변경하면 두 이름 모두 변경을 봄.
변수는 객체에 대한 "참조(포인터)"를 보유한다.

Python은 완전한 참조 의미론을 채택합니다. 단, int, float, str 등 불변(Immutable) 타입은 수정이 불가능하므로 값 의미론처럼 동작하는 것처럼 보입니다.

Immutable — 값 의미론처럼 보이는 이유 
x = 10
y = x       # 같은 10 객체를 가리킴 (참조 의미론)

x = 20      # x를 재바인딩 (10 객체는 그대로, x만 이동)
print(y)    # 10 — y는 영향 없음

# 이는 "복사"처럼 보이지만
# 실제로는 "int는 immutable이라 변경 자체가 불가능"하기 때문

# Mutable에서는 참조 의미론이 직접 드러남
a = [1, 2, 3]
b = a
a.append(4)   # a 객체를 in-place 변경
print(b)      # [1, 2, 3, 4] — b도 같은 객체를 보고 있음
실행 결과:
10 [1, 2, 3, 4]

바인딩 증명 — id()와 is

초급

같은 사람인지 확인하는 법

사람에게는 주민등록번호가 있습니다. 이름이 같아도 번호가 다르면 다른 사람입니다.
Python 객체에도 고유 번호가 있습니다 — id()로 확인합니다.

id()로 같은 객체인지 확인 
장바구니 = ["사과", "바나나", "우유"]
내_장바구니 = 장바구니

print(id(장바구니))         # 예: 140234866532640
print(id(내_장바구니))      # 동일한 번호! — 같은 객체

print(장바구니 is 내_장바구니)   # True — "정말 같은 객체인가?"

# 친구가 같은 내용으로 새 장바구니를 만들면
친구_장바구니 = ["사과", "바나나", "우유"]   # 내용은 같지만
print(id(친구_장바구니))    # 다른 번호 — 다른 객체
print(장바구니 is 친구_장바구니)   # False — 다른 객체
print(장바구니 == 친구_장바구니)   # True — 값은 같음
실행 결과:
140234866532640 140234866532640 True 140234866533760 False True
== 과 is의 차이:
== → 값이 같은가? (내용 비교)
is → 동일한 객체인가? (주소 비교)
중급

작은 정수 캐시와 문자열 인터닝

Python은 자주 사용하는 작은 정수(-5 ~ 256)와 일부 문자열을 미리 만들어 재사용합니다.
이를 인터닝(Interning) 또는 캐싱(Caching)이라 합니다.

작은 정수 캐시 
# 작은 정수: 캐시에서 같은 객체를 반환
a = 100
b = 100
print(a is b)    # True — 같은 캐시 객체

# 큰 정수: REPL에서 한 줄씩 입력할 때 다른 객체
a = 1000
b = 1000
print(a is b)    # CPython 3.12+에서는 REPL에서도 같은 블록 내에서 True가 될 수 있음
                 # (상수 폴딩/인터닝 정책이 버전마다 다름 — 구현 의존적)

# 문자열 인터닝 (식별자처럼 생긴 문자열)
s1 = "hello"
s2 = "hello"
print(s1 is s2)  # 보통 True (인터닝됨)
주의: is로 값을 비교하지 마세요.
a is b는 "같은 객체"를 테스트합니다. 값 비교에는 ==을 사용하세요.
예외: None, True, False는 싱글턴이므로 is None 사용이 권장됩니다.
고급

del — 이름표 제거

del x는 "객체를 삭제"하는 것이 아니라, 이름표 x를 네임스페이스에서 제거하는 것입니다.

del의 정확한 동작 
import sys

x = [1, 2, 3]
y = x

# x가 가리키는 객체의 참조 카운트 확인 (getrefcount는 +1해서 반환)
print(sys.getrefcount(x))   # 3 (x, y, getrefcount 인자)

del x                        # 이름표 x 제거 → 참조 카운트 -1
# print(x)                   # NameError: x가 없음
print(y)                     # [1, 2, 3] — 객체는 여전히 살아있음
                             # y가 아직 가리키고 있으므로
실행 결과:
3 [1, 2, 3]

Python 공식 문서: "문장 del x는 지역 스코프가 참조하는 이름 공간에서 x의 연결을 제거합니다."
객체 자체가 사라지는 것은 참조 카운트가 0이 될 때입니다.

전공자

Part A: DELETE_NAME 바이트코드 + ob_refcnt

del의 바이트코드 
import dis
dis.dis(compile("x = 1; del x", "<s>", "exec"))
실행 결과:
0 0 RESUME 0 1 2 LOAD_CONST 0 (1) 4 STORE_NAME 0 (x) 6 DELETE_NAME 0 (x) 8 RETURN_CONST 1 (None)

DELETE_NAME은 CPython C 코드에서 PyObject_DelItem(f_locals, name)을 호출합니다.
이는 딕셔너리에서 키를 제거하고, 해당 PyObject의 ob_refcnt를 Py_DECREF()로 1 감소시킵니다.
ob_refcnt가 0이 되면 tp_dealloc 함수 포인터를 통해 메모리를 해제합니다.

Mutable + 바인딩 = 수수께끼 해답

초급

이름표 두 개가 같은 것을 가리킬 때

핵심 비유:
방 하나에 문이 두 개 있습니다. 문 A와 문 B.
두 문 다 같은 방으로 연결됩니다.
문 B로 들어가서 방 안에 소파를 추가하면, 문 A로 들어가도 소파가 보입니다.
방(객체)이 바뀐 것이지, 문(이름)이 바뀐 것이 아닙니다.
수수께끼 해답 
할일 = ["수학숙제", "독서", "운동"]   # 할일이라는 이름표를 리스트에 붙임
공유목록 = 할일                       # 공유목록이라는 이름표도 같은 리스트에 붙임
                                     # 지금 할일과 공유목록은 같은 리스트를 가리킴

시간표 = 시간표 + ["과학"]   # 새 리스트 객체를 만들고 시간표를 거기로 재바인딩

# ✅ 해결: 각자 다른 리스트 만들기

b = []
a.append(1)
print(b)   # [] ← 독립적
print(id(a) == id(b))   # False

print()가 str()을 호출하는 방식 — __str__ 프로토콜

print(x)는 내부적으로 x.__str__()을 호출하여 문자열로 변환합니다.
이것이 우리가 직접 만든 클래스도 print()로 출력할 수 있는 이유입니다.

__str__ 프로토콜 
class Point:
    def __init__(self, x, y):
        self.x = x
        self.y = y

    def __str__(self):
        return f"Point({self.x}, {self.y})"

p = Point(3, 4)
print(p)           # Point(3, 4) ← __str__ 호출됨
print(str(p))      # Point(3, 4) ← 동일

# __str__ 없으면?
class Bare:
    pass

b = Bare()
print(b)           # <__main__.Bare object at 0x...> (기본 표현)
실행 결과:
Point(3, 4) Point(3, 4) <__main__.Bare object at 0x7f1234567890>

print()는 Python Data Model의 __str__ 프로토콜을 사용합니다.
모든 built-in 타입은 적절한 __str__을 구현하고 있습니다.

전공자

Part A: builtin_print_impl() → PyObject_Str() → tp_str → sys.stdout.write()

print(x)가 화면에 출력되기까지 CPython 내부에서 거치는 호출 체인을 추적합니다.

print(x)의 바이트코드 
import dis
dis.dis(compile("print(x)", "<s>", "exec"))
실행 결과:
0 0 RESUME 0 1 2 PUSH_NULL 4 LOAD_NAME 0 (print) 6 LOAD_NAME 1 (x) 8 CALL 1 16 POP_TOP 18 RETURN_CONST 0 (None)

CALL 1이 실행될 때 CPython C 코드(Python/bltinmodule.c)의 builtin_print_impl()이 호출됩니다. 이 함수의 핵심 동작:

  1. 각 인자 객체에 대해 PyObject_Str(obj) 호출
  2. PyObject_Str()은 obj->ob_type->tp_str(obj)를 호출
    → 이것이 Python 레벨의 obj.__str__()에 해당
  3. tp_str이 NULL이면 tp_repr로 fallback (__repr__)
  4. 반환된 str 객체를 file.write()로 출력 (file 기본값: sys.stdout)
  5. 각 객체 사이에 sep을 write, 마지막에 end를 write
sys.stdout의 실제 구조

print(type(sys.stdout))
# <class '_io.TextIOWrapper'>

# sys.stdout은 3겹 래퍼 구조:
# TextIOWrapper     ← 텍스트 인코딩/디코딩 (str ↔ bytes)
#   └ BufferedWriter ← 쓰기 버퍼링
#       └ FileIO      ← 실제 OS 파일 디스크립터 (fd=1, stdout)

네임스페이스 & LEGB 규칙

중급

이름표를 어디서 찾는가?

x라고 쓰면 Python은 어디서 x를 찾을까요?
Python은 4단계 순서로 이름을 찾습니다.

LEGB 규칙 (Local → Enclosing → Global → Built-in)

Local — 현재 함수 안
Enclosing — 바깥쪽 함수 안 (중첩 함수일 때)
Global — 모듈 전체
Built-in — Python 기본 내장 (print, len 등)
LEGB 탐색 순서 확인 
x = "Global"           # Global(G)

def outer():
    x = "Enclosing"    # Enclosing(E)

    def inner():
        x = "Local"    # Local(L)
        print(x)       # "Local" ← L에서 발견

    inner()
    print(x)           # "Enclosing" ← L에 없으니 E에서 발견

outer()
print(x)               # "Global" ← G에서 발견

# Built-in 예시
print(len([1,2,3]))    # len은 B(Built-in)에서 발견
실행 결과:
Local Enclosing Global 3
고급

global과 nonlocal — 바깥 스코프에 쓰기

Python 공식 문서: "global이나 nonlocal 문이 없을 때 — 이름에 대입하면 항상 가장 내부의 스코프로 간다."

global과 nonlocal의 차이 
count = 0   # 전역 변수

def increment_wrong():
    count = count + 1   # UnboundLocalError!
    # Python이 count를 지역변수로 인식 → 초기화 전 참조 오류

def increment_right():
    global count        # 전역 count를 쓰겠다 선언
    count = count + 1

increment_right()
print(count)            # 1

# nonlocal: 바깥 함수의 변수에 쓰기
def make_counter():
    n = 0
    def counter():
        nonlocal n      # 바깥 함수 스코프의 n을 쓰겠다
        n += 1
        return n
    return counter

c = make_counter()
print(c())    # 1
print(c())    # 2
print(c())    # 3
실행 결과:
1 1 2 3
🚨 함정 — 함수 안에서 전역변수를 수정하려면?
점수 = 100

def 감점():
    점수 = 점수 - 10   # ❌ UnboundLocalError!
    return 점수

# 왜? Python은 함수 본문 전체를 스캔해서
# 대입문(=)이 있는 이름을 지역변수로 결정합니다.
# "점수 = 점수 - 10"에서 왼쪽의 = 때문에
# 점수가 지역변수가 되었는데, 오른쪽에서 아직 값이 없는
# 지역변수 점수를 읽으려 해서 에러!

# ✅ 해결법 1: global 선언
def 감점_수정():
    global 점수
    점수 = 점수 - 10
    return 점수
왜 global은 나쁜 습관인가?
전역 변수를 함수 안에서 수정하면 코드를 읽는 사람이 "이 함수가 어떤 외부 상태를 바꾸는가"를 추적해야 합니다. 함수형 프로그래밍에서는 함수가 입력에만 의존하고 외부 상태를 바꾸지 않는 것을 권장합니다(순수 함수).

Part A: 네임스페이스의 CPython 구현 — f_locals, f_globals

CPython의 실행 프레임(PyFrameObject)은 두 개의 딕셔너리를 가집니다:

프레임의 네임스페이스 구조 
import sys

x = 10        # 전역 네임스페이스

def show_namespaces():
    y = 20    # 지역 네임스페이스
    frame = sys._getframe()   # 현재 실행 프레임

    print("지역(locals):", frame.f_locals)
    print("전역(globals) 일부:", {k: v for k, v in frame.f_globals.items()
                                   if not k.startswith('__')})

show_namespaces()
실행 결과:
지역(locals): {'y': 20, 'frame': <frame object at 0x...>} 전역(globals) 일부: {'x': 10, 'show_namespaces': <function ...>}

LOAD_NAME은 f_locals → f_globals → f_builtins 순으로 탐색합니다.
LOAD_FAST는 f_localsplus C 배열을 직접 인덱스로 접근합니다(딕셔너리 우회).
LOAD_GLOBAL(함수 내에서 전역 접근)은 f_globals를 먼저 탐색하고, 없으면 f_builtins로 fallback합니다.

Python 3.12+ 변경:
LOAD_FAST_CHECK가 추가되어 지역 변수 초기화 여부를 더 효율적으로 확인합니다.
LOAD_SUPER_ATTR도 3.12+에서 새로 추가되었습니다.

Part B — 변수 바인딩과 메모리 관리: 참조 카운팅과 GC

CPython의 메모리 관리는 참조 카운팅(Reference Counting)이 기반입니다. 모든 PyObject는 ob_refcnt 필드를 가지며, 바인딩될 때 증가하고 해제될 때 감소합니다.

참조 카운팅과 순환 참조 
import sys
import gc

x = [1, 2, 3]
print(sys.getrefcount(x))   # 2 (x 바인딩 1 + getrefcount 인자 1)

y = x
print(sys.getrefcount(x))   # 3 (y도 같은 객체를 참조)

del y
print(sys.getrefcount(x))   # 2 (y 해제로 감소)

# 순환 참조: 참조 카운팅만으로는 해결 불가
a = []
b = [a]
a.append(b)    # a → b → a (순환)
# del a, del b 해도 서로 참조하므로 refcnt가 0이 되지 않음
# → 세대별 GC (gc 모듈)가 처리

gc.collect()   # 수동으로 GC 실행 (순환 참조 해제)
print("GC 완료")
실행 결과:
2 3 2 GC 완료
CPython GC 3세대:
• 0세대: 새로 생성된 객체 (자주 수거)
• 1세대: 0세대 GC 후 살아남은 객체
• 2세대: 1세대 GC 후 살아남은 객체 (드물게 수거)
순환 참조가 없다면 CPython은 대부분 참조 카운팅만으로 즉시 메모리를 해제합니다.

AI가 자주 만드는 스코프 버그

    funcs.append(lambda: i)   # AI가 자주 만드는 패턴

Part A: 클로저의 CPython 구현 — co_freevars와 cell 객체

클로저(Closure)는 함수가 자신이 정의된 스코프의 변수를 캡처하는 메커니즘입니다.

클로저의 내부 구조 
import dis

def outer():
    x = 10
    def inner():
        return x    # outer의 x를 캡처
    return inner

f = outer()
print(f())          # 10

# 클로저 변수 확인
print(f.__code__.co_freevars)    # ('x',) — 캡처된 이름
print(f.__closure__)             # (<cell at 0x...>,)
print(f.__closure__[0].cell_contents)   # 10 — 실제 값

# dis로 LOAD_DEREF 확인
dis.dis(f)
실행 결과:
10 ('x',) (<cell at 0x7f1234567890>,) 10 0 COPY_FREE_VARS 1 2 RESUME 0 4 LOAD_DEREF 0 (x) 6 RETURN_VALUE

LOAD_DEREF: 클로저 변수는 cell 객체를 통해 간접 참조됩니다.
이것이 "late binding closure" 버그의 원인입니다: 모든 lambda가 같은 cell 객체를 공유하며, 루프가 진행될 때마다 cell 내부의 참조가 i의 최신 값으로 갱신됩니다. 루프가 끝나면 cell이 가리키는 객체는 마지막 값인 2입니다.

전공자

Part A — LOAD_FAST vs LOAD_DEREF vs LOAD_GLOBAL: 바이트코드로 보는 LEGB

파이썬의 LEGB 규칙은 바이트코드 레벨에서 서로 다른 명령어로 구현됩니다. 각 스코프의 변수 접근이 어떤 바이트코드를 사용하는지 직접 확인합니다.

LEGB별 바이트코드 확인 
import dis

x = "global"   # Global

def outer():
    y = "enclosing"   # Enclosing

    def inner():
        z = "local"   # Local
        print(x)      # Global → LOAD_GLOBAL
        print(y)      # Enclosing → LOAD_DEREF (클로저 변수)
        print(z)      # Local → LOAD_FAST (가장 빠름)

    return inner

inner_fn = outer()

print("=== inner 함수 바이트코드 ===")
dis.dis(inner_fn)
print()
print("=== 클로저 변수 확인 ===")
print(inner_fn.__code__.co_freevars)   # ('y',) ← y가 클로저 변수
print(inner_fn.__code__.co_varnames)   # ('z',) ← z는 로컬 변수
print(inner_fn.__closure__)            # (<cell at 0x...>,)

Part B — 환경 모델(Environment Model) vs 치환 모델(Substitution Model)

컴퓨터 과학에서 프로그램의 실행을 설명하는 두 가지 모델이 있습니다. SICP(Structure and Interpretation of Computer Programs)에서 소개한 개념입니다.

치환 모델(Substitution Model): 함수 호출 시 인자를 함수 본체에 대입합니다.
간단하지만 변수 수정(mutation)과 클로저를 설명할 수 없습니다.

환경 모델(Environment Model): 각 스코프는 이름→값 매핑(환경)을 가지며,
부모 환경을 참조하는 체인을 형성합니다. Python의 LEGB 규칙이 이 모델입니다.
환경 모델로 클로저 이해하기 
import inspect

# 환경 모델로 클로저 이해하기
def make_adder(n):
    # 환경 E1: {n: 5} (부모: 전역 환경)
    def adder(x):
        # 환경 E2: {x: 3} (부모: E1)
        return n + x   # n을 찾으러 E1으로 올라감 → 5
    return adder

add5 = make_adder(5)
# add5의 클로저: E1 환경({n: 5})을 포함
print(add5(3))   # 8 (E2에서 x=3, E1에서 n=5)
print(add5(7))   # 12 (E2에서 x=7, E1에서 n=5)

# 두 클로저는 독립적인 환경을 가짐
add10 = make_adder(10)
print(add5(1))   # 6 (E1: n=5)
print(add10(1))  # 11 (E1': n=10) ← 다른 환경!

LEGB 규칙

⚡

STORE_FAST

함수 지역 변수는 딕셔너리가 아닌 C 배열로 관리. 전역보다 빠른 이유.

용어 사전 (26개)

Binding (바인딩)
이름(변수)을 객체에 연결하는 것. x = 1은 이름 x를 정수 객체 1에 바인딩.
Rebinding (재바인딩)
이미 바인딩된 이름을 다른 객체에 연결하는 것. x = 2는 x를 2로 재바인딩.
Reference (참조)
객체를 가리키는 포인터. Python에서 변수는 객체에 대한 참조를 보유.
Reference Count (참조 카운트)
객체를 가리키는 참조의 수. 0이 되면 메모리 해제. ob_refcnt 필드.
id()
객체의 고유 정수 식별자 반환. CPython에서 메모리 주소. 객체 생존 중 불변.
is 연산자
두 이름이 동일한 객체를 가리키는지 확인. id()가 같은지와 동일. 값 비교엔 ==.
Namespace (네임스페이스)
이름에서 객체로 가는 매핑(딕셔너리). 모듈 전역, 함수 지역, 내장 이름공간 등.
Scope (스코프)
이름 공간을 직접 접근할 수 있는 코드 영역. 텍스트적으로(정적으로) 결정됨.
LEGB 규칙
이름 탐색 순서: Local → Enclosing → Global → Built-in. Python 스코프 해석 규칙.
Local Scope (지역 스코프)
현재 함수 안에서 정의된 이름들의 범위. LEGB의 L.
Enclosing Scope (외부 함수 스코프)
중첩 함수에서 바깥쪽 함수의 스코프. LEGB의 E. nonlocal로 접근.
Global Scope (전역 스코프)
모듈 수준에서 정의된 이름들의 범위. LEGB의 G. global로 함수에서 접근.
Built-in Scope (내장 스코프)
print, len, int 등 Python이 기본 제공하는 이름들. LEGB의 B.
global 키워드
함수 내에서 모듈 전역 변수에 대입하기 위한 선언. 없으면 지역 변수로 취급.
nonlocal 키워드
중첩 함수에서 바깥 함수 변수에 대입하기 위한 선언.
Shallow Copy (얕은 복사)
최상위 컨테이너만 새로 만들고 내부 객체는 같은 참조 공유. .copy(), [:].
Deep Copy (깊은 복사)
모든 중첩 객체를 재귀적으로 복사. copy.deepcopy(). 완전히 독립된 복사본.
In-place Modification (제자리 수정)
기존 객체의 내용을 직접 수정. list.append(), list.sort() 등. 객체 id 변하지 않음.
Unpacking (언패킹)
시퀀스/이터러블의 원소를 여러 이름에 동시 대입. a, b = 1, 2.
STORE_NAME
CPython 바이트코드. 모듈/클래스 스코프에서 이름을 딕셔너리에 저장.
STORE_FAST
CPython 바이트코드. 함수 스코프에서 이름을 C 배열에 저장. 딕셔너리보다 빠름.
Closure (클로저)
함수가 자신이 정의된 스코프의 변수를 캡처하는 것. cell 객체를 통해 간접 참조.
Mutable (가변)
생성 후 내용을 변경할 수 있는 객체. list, dict, set 등. lst.append()로 제자리 수정 가능
Immutable (불변)
생성 후 내용을 변경할 수 없는 객체. int, str, tuple 등. 수정하려면 새 객체를 만들어야 함
Aliasing (별칭)
같은 객체를 여러 이름이 가리키는 상태. b = a 후 a와 b는 같은 객체의 별칭
UnboundLocalError
함수 내에서 지역변수에 값을 할당하기 전에 참조할 때 발생하는 오류. 컴파일 시점에 지역변수로 결정됨

연산자와 표현식 — 컴퓨터는 어떻게 계산하는가?

이름표와 바인딩을 배웠다면, 다음 주에는 "값들을 어떻게 조합하는가"를 배웁니다.

  • 왜 ==과 =은 다른가? — 대입 vs 비교
  • 연산자 우선순위: 2 + 3 * 4는 왜 20이 아닌 14인가?
