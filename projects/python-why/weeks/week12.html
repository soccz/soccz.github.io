지난 주 제너레이터를 만들었습니다. 그때 for val in gen이 동작한 이유는 제너레이터가 __iter__와 __next__를 구현했기 때문입니다. 이것이 던더 메서드(dunder method)의 핵심입니다 — Python 인터프리터와의 약속. 

# 우리가 쓰는 코드
v1 = Vector(3, 4)
v2 = Vector(1, 2)
v3 = v1 + v2     # 이게 어떻게 동작하는가?

class Point:
    def __init__(self, x, y):

p1 = Point(3, 4)
p2 = Point(1, 2)

# 두 점을 더하고 싶은데...

int나 str은 +, ==, print가 자연스럽게 동작합니다. 우리 클래스는 왜 안 될까요? 답: Python에게 "어떻게 더하고, 비교하고, 출력할지" 알려주지 않았기 때문입니다. 던더 메서드로 알려주면 됩니다.

연산자는 메서드 호출이다

비유: 번역기
Python 인터프리터는 a + b를 보면 번역합니다 — "a.__add__(b)를 호출하라."
len(a)는 "a.__len__()을 호출하라."
a[0]는 "a.__getitem__(0)을 호출하라."
던더 메서드는 이 번역 규칙에 따른 약속된 이름입니다. 
class Box:
    def __init__(self, items):
        self.items = items

    def __len__(self):
        return len(self.items)

    def __getitem__(self, index):
        return self.items[index]

    def __repr__(self):
        return f"Box({self.items})"

box = Box([1, 2, 3, 4, 5])

# 내장 함수처럼 동작
print(len(box))        # 5  ← __len__ 호출
print(box[2])          # 3  ← __getitem__ 호출
print(box)             # Box([1, 2, 3, 4, 5])  ← __repr__ 호출

# for문도 가능 (__getitem__이 있으면 이터러블처럼 동작)
for item in box:
    print(item, end=" ")
# 1 2 3 4 5
던더는 직접 호출하지 않습니다 
box.__len__()   # 동작하긴 하지만 — 이렇게 쓰지 않는다
len(box)        # 올바른 방법

# 이유: len()은 단순 호출이 아님
# 내부적으로 타입 확인, 최적화, 오버로드 처리 등을 수행
중급

주요 던더 메서드 분류

class FullDemo:
    # --- 객체 생성 ---
    def __init__(self, value):         # 초기화
        self.value = value

    # --- 문자열 표현 ---
    def __repr__(self):                # 개발자용
        return f"FullDemo({self.value!r})"
    def __str__(self):                 # 사용자용
        return f"값: {self.value}"

    # --- 비교 연산자 ---
    def __eq__(self, other):           # ==
        return self.value == other.value
    def __lt__(self, other):           # <
        return self.value < other.value
    def __le__(self, other):           # <=
        return self.value <= other.value

    # --- 산술 연산자 ---
    def __add__(self, other):          # +
        return FullDemo(self.value + other.value)
    def __mul__(self, n):              # * (숫자와)
        return FullDemo(self.value * n)

    # --- 컨테이너 프로토콜 ---
    def __len__(self):                 # len()
        return len(str(self.value))
    def __contains__(self, item):      # in 연산자
        return item in str(self.value)

    # --- 불리언 변환 ---
    def __bool__(self):                # bool(), if에서
        return bool(self.value)

a = FullDemo(10)
b = FullDemo(20)
print(a + b)           # FullDemo(30)
print(a < b)           # True
print("1" in a)        # True (str(10)에 "1" 포함)

반영 메서드 (Reflected Methods) — r-접두어

a + b에서 a.__add__(b)가 NotImplemented를 반환하면, Python은 b.__radd__(a)를 시도합니다. 

class Currency:
    def __init__(self, amount):
        self.amount = amount

    def __add__(self, other):
        if isinstance(other, (int, float)):
            return Currency(self.amount + other)
        if isinstance(other, Currency):
            return Currency(self.amount + other.amount)
        return NotImplemented   # "모르겠다, 오른쪽한테 물어봐"

    def __radd__(self, other):  # other + self 일 때 호출
        return self.__add__(other)

    def __repr__(self):
        return f"Currency({self.amount})"

c = Currency(100)
print(c + 50)     # Currency(150)  ← __add__
print(50 + c)     # Currency(150)  ← int.__add__(c) → NotImplemented → c.__radd__(50)

# sum()이 동작하려면 __radd__ 필요
print(sum([Currency(10), Currency(20), Currency(30)]))
# sum은 내부적으로 0 + item을 반복 — 첫 항이 0 + Currency(10) = __radd__ 호출
NotImplemented vs NotImplementedError
NotImplemented: 반환값 — "이 타입은 처리 못해, 오른쪽 피연산자에게 물어봐"
NotImplementedError: 예외 — "이 메서드는 서브클래스가 구현해야 함"
반영 메서드에서는 반드시 NotImplemented를 반환해야 합니다.

Part A: CPython — 던더 메서드 조회 방식

CPython은 던더 메서드를 일반 속성 조회(getattr)로 찾지 않습니다. 타입 객체의 C 함수 포인터 슬롯(tp_* 필드)을 직접 참조합니다. 

import dis

class Slow:
    def __len__(self):
        return 42

s = Slow()
# len()은 type(s).__len__를 찾아 tp_as_sequence->sq_length를 캐싱
# 인스턴스 딕셔너리는 조회하지 않음

# 따라서 인스턴스에 직접 __len__을 붙여도 len()이 그걸 호출하지 않음
s.__len__ = lambda: 99
print(len(s))    # 42  ← 인스턴스의 __len__ 무시, 클래스의 것을 씀
print(s.__len__())  # 99  ← 직접 호출하면 인스턴스 것이 나옴

이것이 Python에서 특수 메서드는 클래스에 정의해야 하는 이유입니다. 인스턴스에 붙여도 내장 연산자는 무시합니다.

Part B: functools.total_ordering

6개의 비교 메서드(__lt__, __le__, __gt__, __ge__, __eq__, __ne__) 중 __eq__와 하나만 구현하면 나머지를 자동 생성해주는 데코레이터입니다. 

from functools import total_ordering

@total_ordering
class Temperature:
    def __init__(self, celsius):
        self.celsius = celsius

    def __eq__(self, other):
        return self.celsius == other.celsius

    def __lt__(self, other):       # 이 두 개만 구현
        return self.celsius < other.celsius

    def __repr__(self):
        return f"{self.celsius}°C"

t1 = Temperature(20)
t2 = Temperature(30)
print(t1 < t2)    # True  — __lt__ 직접
print(t1 > t2)    # False — total_ordering이 __gt__ 자동 생성
print(t1 <= t2)   # True  — total_ordering이 __le__ 자동 생성
print(sorted([Temperature(30), Temperature(10), Temperature(20)]))
# [10°C, 20°C, 30°C]

__new__ vs __init__ — 공장과 인테리어

초급

__init__은 생성자가 아닙니다

비유: 공장과 인테리어
__new__: 공장에서 빈 집의 뼈대(콘크리트 구조)를 짓습니다. 메모리에 새 객체의 틀을 만듭니다.
__init__: 완성된 뼈대 안에 가구를 배치하고 색을 칠합니다. 이미 만들어진 인스턴스에 속성을 채웁니다.
생성자(constructor)는 __new__입니다. __init__은 초기화자(initializer)입니다. 
class Dog:
    def __init__(self, name, breed):
        # 이 시점에서 self는 이미 만들어진 빈 Dog 인스턴스
        # __new__가 이미 메모리를 할당함
        self.name = name
        self.breed = breed

    def bark(self):
        print(f"{self.name}: 왈왈!")

# Dog("바둑이", "진돗개")를 호출하면:
# 1. Dog.__new__(Dog, "바둑이", "진돗개") 실행 → 빈 Dog 객체 생성
# 2. Dog.__init__(새_객체, "바둑이", "진돗개") 실행 → 속성 채움

dog1 = Dog("바둑이", "진돗개")
dog1.bark()  # 바둑이: 왈왈!

대부분의 경우 __new__를 직접 구현할 필요 없습니다. object.__new__가 기본 동작을 합니다.

중급

__new__를 오버라이드해야 할 때

# 싱글톤 패턴 — 인스턴스가 하나뿐이어야 할 때
class Singleton:
    _instance = None   # 클래스 변수

    def __new__(cls):
        if cls._instance is None:
            cls._instance = super().__new__(cls)  # 딱 한 번만 생성
        return cls._instance    # 항상 같은 객체 반환

a = Singleton()
b = Singleton()
print(a is b)        # True — 같은 객체
print(id(a), id(b))  # 같은 주소

# __new__는 인스턴스를 반환해야 함
# 반환하지 않으면 __init__이 호출되지 않음
불변 타입 서브클래싱 시 __new__ 필수 
# int, str, tuple은 불변 — __init__에서 값 변경 불가
# __new__에서 값을 설정해야 함
class PositiveInt(int):
    def __new__(cls, value):
        if value <= 0:
            raise ValueError(f"{value}는 양수가 아닙니다")
        return super().__new__(cls, value)  # int 생성 시 값 전달

p = PositiveInt(5)
print(p)           # 5
print(p + 3)       # 8  (int처럼 동작)
PositiveInt(-1)    # ValueError
고급

__new__와 __init__의 인자 전달 규칙

class Verbose:
    def __new__(cls, *args, **kwargs):
        print(f"__new__(cls={cls.__name__}, args={args}, kwargs={kwargs})")
        instance = super().__new__(cls)
        return instance

    def __init__(self, value, label="default"):
        print(f"__init__(self={id(self):#x}, value={value}, label={label})")
        self.value = value
        self.label = label

obj = Verbose(42, label="test")
# __new__(cls=Verbose, args=(42,), kwargs={'label': 'test'})
# __init__(self=0x..., value=42, label=test)

__new__와 __init__은 같은 인자를 받습니다. __new__가 현재 클래스와 다른 타입의 인스턴스를 반환하면 __init__은 호출되지 않습니다. 

class Factory:
    def __new__(cls, type_name):
        if type_name == "int":
            return 42       # int 반환 → __init__ 호출 안 됨
        elif type_name == "str":
            return "hello"  # str 반환 → __init__ 호출 안 됨
        return super().__new__(cls)   # Factory 반환 → __init__ 호출됨

    def __init__(self, type_name):
        print(f"__init__ 호출됨: type_name={type_name}")

print(Factory("int"))    # 42  (__init__ 출력 없음)
print(Factory("str"))    # hello (__init__ 출력 없음)
obj = Factory("other")  # __init__ 호출됨: type_name=other
print(type(obj))         # <class '__main__.Factory'>

Part B: 왜 __init_subclass__가 필요했는가

Python 3.6 이전에는 "서브클래스가 만들어질 때 자동으로 뭔가를 하고 싶으면" 메타클래스를 써야 했습니다. 메타클래스는 강력하지만 복잡하고, 다중 상속 시 메타클래스 충돌 문제가 있었습니다.

__init_subclass__(PEP 487)는 이 문제를 해결합니다 — 메타클래스 없이 "서브클래스 등록" 같은 간단한 훅을 구현할 수 있습니다. "간단한 일에는 간단한 도구를" — Python의 설계 원칙입니다. 

# Python 3.6+ — 클래스가 서브클래싱될 때 자동 호출
class Plugin:
    _registry = {}

    def __init_subclass__(cls, plugin_name=None, **kwargs):
        super().__init_subclass__(**kwargs)
        if plugin_name:
            Plugin._registry[plugin_name] = cls
            print(f"플러그인 등록: {plugin_name} → {cls.__name__}")

class AudioPlugin(Plugin, plugin_name="audio"):
    def process(self): return "오디오 처리"

class VideoPlugin(Plugin, plugin_name="video"):
    def process(self): return "비디오 처리"

# 등록 결과:
# 플러그인 등록: audio → AudioPlugin
# 플러그인 등록: video → VideoPlugin

print(Plugin._registry)
# {'audio': <class 'AudioPlugin'>, 'video': <class 'VideoPlugin'>}

# 플러그인 동적 사용
p = Plugin._registry["audio"]()
print(p.process())  # 오디오 처리

__repr__ vs __str__ — 개발자용 vs 사용자용

초급

두 가지 얼굴

비유: 명함과 자기소개
__repr__: 개발자용 명함 — "Point(x=3, y=4)" — 복사해서 코드에 붙여넣을 수 있는 형식.
__str__: 사용자용 자기소개 — "(3, 4)" — 보기 좋고 이해하기 쉬운 형식.
Python의 원칙: repr()은 eval()로 복원 가능해야 한다. 
class Point:
    def __init__(self, x, y):
        self.x, self.y = x, y

    def __repr__(self):
        return f"Point({self.x}, {self.y})"   # 개발자용 — 복원 가능

    def __str__(self):
        return f"({self.x}, {self.y})"        # 사용자용 — 보기 좋음

p = Point(3, 4)
print(repr(p))    # Point(3, 4)  ← __repr__
print(str(p))     # (3, 4)        ← __str__
print(p)          # (3, 4)        ← print()는 __str__ 우선
print(f"{p}")     # (3, 4)        ← f-string도 __str__
print(f"{p!r}")   # Point(3, 4)   ← !r은 __repr__ 강제

# 컨테이너 안에서는 항상 __repr__
points = [Point(1, 2), Point(3, 4)]
print(points)     # [Point(1, 2), Point(3, 4)]  ← 리스트는 repr 사용
__repr__만 정의하면 __str__도 됩니다
__str__이 없으면 Python은 __repr__으로 대체합니다.
반대는 아닙니다 — __repr__은 항상 구현해야 합니다.

class Money:
    def __init__(self, amount, currency="KRW"):
        self.amount = amount
        self.currency = currency

    def __repr__(self):
        return f"Money({self.amount}, {self.currency!r})"

    def __str__(self):
        return f"{self.amount:,} {self.currency}"

    def __format__(self, spec):
        if spec == "usd":
            return f"${self.amount / 1300:,.2f}"
        elif spec == "plain":
            return f"{self.amount}"
        return str(self)   # 기본은 __str__

m = Money(130000)
print(str(m))          # 130,000 KRW
print(repr(m))         # Money(130000, 'KRW')
print(f"{m}")          # 130,000 KRW
print(f"{m:usd}")      # $100.00
print(f"{m:plain}")    # 130000

__eq__ 오버라이드 시 __hash__가 None이 되는 이유

Python은 __eq__를 오버라이드하면 __hash__를 None으로 설정합니다 — "안전하게 해시 불가 타입으로."

    def __init__(self, x, y):
        self.x, self.y = x, y
    def __eq__(self, other):
        return self.x == other.x and self.y == other.y
    # __hash__를 정의 안 함 → Python이 None으로 설정

    hash(p)             # TypeError: unhashable type: 'Point'
except TypeError as e:
    print(e)

# 딕셔너리 키나 세트 원소로 쓰려면 __hash__도 구현해야 함
class HashablePoint:
    def __init__(self, x, y):
        self.x, self.y = x, y
    def __eq__(self, other):
        return self.x == other.x and self.y == other.y
    def __hash__(self):
        return hash((self.x, self.y))   # 튜플의 hash 활용

p1 = HashablePoint(1, 2)
p2 = HashablePoint(1, 2)
s = {p1, p2}
print(len(s))   # 1 — 같은 원소로 인식

print(d[p2])    # origin에서 가까운 점 (p2로 조회해도 됨)

class BadHash:
    def __init__(self, value):
        self.value = value
    def __eq__(self, other):
        return self.value == other.value
    def __hash__(self):
        return 1  # 모두 같은 해시값 → 딕셔너리가 O(n)으로 느려짐

# 좋은 __hash__ — 고른 분포
class GoodHash:
    def __init__(self, a, b, c):
        self.a, self.b, self.c = a, b, c
    def __eq__(self, other):
        return (self.a, self.b, self.c) == (other.a, other.b, other.c)
    def __hash__(self):
        return hash((self.a, self.b, self.c))  # 튜플은 각 원소를 혼합

super()는 부모가 아니라 "다음 차례"

비유: 회사 결재선
상사 A에게 결재 요청했는데 A가 "나는 모르겠어, 위에 B한테 물어봐"합니다.
B는 "나도 위에 C한테 물어봐"합니다. 이것이 super()입니다.
단순히 "부모"가 아니라 MRO(Method Resolution Order, 메서드 탐색 순서) 상의 다음 클래스에게 위임합니다. 
class A:
    def hello(self):
        print("A.hello")

class B(A):
    def hello(self):
        print("B.hello")
        super().hello()    # "다음 클래스(A)에게 위임"

class C(A):
    def hello(self):
        print("C.hello")
        super().hello()    # "다음 클래스(A)에게 위임"

class D(B, C):             # 다중 상속
    def hello(self):
        print("D.hello")
        super().hello()    # "다음 클래스(B)에게 위임"

# MRO 확인
print([cls.__name__ for cls in D.mro()])
# ['D', 'B', 'C', 'A', 'object']

D().hello()
# D.hello
# B.hello   ← super()가 B → C를 알고 있음 (A가 아님!)
# C.hello   ← super()가 C → A를 알고 있음
# A.hello
super()가 중요한 이유: 다이아몬드 문제 해결
D → B, C → A 구조(다이아몬드)에서 A.hello가 두 번 호출되는 문제를 MRO + super()가 해결합니다.
MRO가 각 클래스를 정확히 한 번만 호출하도록 순서를 정해줍니다.
함정: super().__init__() 호출 누락 
# 될 것 같은데 안 되는 코드
class Animal:
    def __init__(self, name):
        self.name = name

class Dog(Animal):
    def __init__(self, name, breed):
        # super().__init__(name)을 안 불렀다!
        self.breed = breed

    def __init__(self, name, breed):
        super().__init__(name)    # 부모 초기화를 명시적으로 호출
        self.breed = breed

class Base:
    def __init__(self, **kwargs):
        super().__init__(**kwargs)    # object.__init__ 으로 이어짐

class Mixin1(Base):
    def __init__(self, a=None, **kwargs):
        print(f"Mixin1.__init__(a={a})")
        super().__init__(**kwargs)   # 다음 MRO로 넘김
        self.a = a

class Mixin2(Base):
    def __init__(self, b=None, **kwargs):
        print(f"Mixin2.__init__(b={b})")
        super().__init__(**kwargs)
        self.b = b

class Combined(Mixin1, Mixin2):
    def __init__(self, a=None, b=None):
        super().__init__(a=a, b=b)

obj = Combined(a=1, b=2)
# Mixin1.__init__(a=1)
# Mixin2.__init__(b=2)
print(obj.a, obj.b)  # 1 2

C3 선형화 알고리즘 — MRO 계산 방식

MRO는 C3 알고리즘으로 계산됩니다. 규칙은 단순합니다:

  1. 자식 클래스는 부모보다 먼저 온다
  2. 부모 클래스의 순서가 유지된다
  3. 위 규칙이 충돌하면 TypeError
# C3 수동 계산 예시
# class D(B, C): B는 A를 상속, C는 A를 상속
# L[D] = D + merge(L[B], L[C], [B, C])
# L[B] = [B, A, object]
# L[C] = [C, A, object]
# merge([B,A,object], [C,A,object], [B,C]):
#   → B가 어디서도 꼬리에 없음 → B 선택
# merge([A,object], [C,A,object], [C]):
#   → A가 [C,A,object]의 꼬리에 있음 (C가 앞에 있으므로) → C 선택
# merge([A,object], [A,object], []):
#   → A → object
# 결과: D → B → C → A → object

# 불가능한 MRO
class X: pass
class Y: pass
class A(X, Y): pass
class B(Y, X): pass
try:
    class C(A, B): pass  # X와 Y의 순서가 A, B에서 충돌
except TypeError as e:
    print(e)  # Cannot create a consistent method resolution order...

Part B: 왜 믹스인(Mixin)이 다중 상속의 올바른 사용법인가

다중 상속은 위험합니다 — 다이아몬드 문제, 이름 충돌, 복잡한 MRO. 그래서 Java는 다중 상속을 금지했고, C++은 virtual inheritance라는 복잡한 메커니즘을 도입했습니다.

Python의 접근법: 다중 상속은 허용하되, "올바른 사용 패턴"을 관례로 정합니다. 그것이 믹스인(Mixin)입니다 — 단일 기능만 제공하는 작은 클래스를 조합합니다. 믹스인은 단독으로 인스턴스화하지 않고, 다른 클래스에 "능력"을 추가하는 용도로만 씁니다.

이 패턴이 안전한 이유: 각 믹스인이 독립적인 메서드만 가지므로 이름 충돌이 거의 없고, MRO가 단순하게 유지됩니다. Django의 View 시스템(LoginRequiredMixin, PermissionRequiredMixin)이 이 패턴의 실전 예입니다. 

# 믹스인: 단일 책임의 작은 클래스들을 조합
class JSONMixin:
    def to_json(self):
        import json
        return json.dumps(self.__dict__)

class LogMixin:
    def log(self, msg):
        print(f"[{type(self).__name__}] {msg}")

class ValidationMixin:
    def validate(self):
        for key, value in self.__dict__.items():
            if value is None:
                raise ValueError(f"{key}는 None이 될 수 없습니다")
        return True

class User(JSONMixin, LogMixin, ValidationMixin):
    def __init__(self, name, email):
        self.name = name
        self.email = email

u = User("김민준", "minjun@example.com")
print(u.to_json())    # {"name": "김민준", "email": "minjun@example.com"}
u.log("사용자 생성됨")   # [User] 사용자 생성됨
u.validate()           # 통과

# MRO: User → JSONMixin → LogMixin → ValidationMixin → object
print([c.__name__ for c in User.mro()])

from contextlib import contextmanager

@contextmanager
def vector_ops_tracker():
    """벡터 연산 횟수를 추적하는 컨텍스트 매니저"""
    ops = {"add": 0, "mul": 0, "abs": 0}
    original_add = Vector.__add__
    original_mul = Vector.__mul__
    original_abs = Vector.__abs__

    def tracked_add(self, other):
        ops["add"] += 1
        return original_add(self, other)

    def tracked_mul(self, scalar):
        ops["mul"] += 1
        return original_mul(self, scalar)

    def tracked_abs(self):
        ops["abs"] += 1
        return original_abs(self)

    Vector.__add__ = tracked_add
    Vector.__mul__ = tracked_mul
    Vector.__abs__ = tracked_abs

    try:
        yield ops
    finally:
        Vector.__add__ = original_add
        Vector.__mul__ = original_mul
        Vector.__abs__ = original_abs

with vector_ops_tracker() as ops:
    v = Vector(3, 4)
    result = abs(v * 2 + Vector(1, 0))
    print(f"결과: {result:.2f}")

print(f"연산 횟수: {ops}")

            return NotImplemented
        return self.celsius <= other.celsius

            return NotImplemented
        return self.celsius > other.celsius

            return NotImplemented
        return self.celsius == other.celsius

        if not isinstance(other, TemperatureSimple):
            return NotImplemented
        return self.celsius < other.celsius

    def __repr__(self):
        return f"TemperatureSimple({self.celsius})"

문제 12. C3 선형화 알고리즘을 직접 구현하세요. 입력으로 클래스와 부모 클래스 리스트를 받아 MRO를 반환합니다.

정답 보기 
def c3_merge(sequences):
    result = []
    while True:
        # 비어있지 않은 시퀀스만
        sequences = [s for s in sequences if s]
        if not sequences:
            return result
        # 각 시퀀스의 헤드 후보 중 다른 시퀀스 꼬리에 없는 것 선택
        for seq in sequences:
            candidate = seq[0]
            if not any(candidate in s[1:] for s in sequences):
                result.append(candidate)
                # 선택된 후보를 모든 시퀀스에서 제거
                for s in sequences:
                    if s[0] == candidate:
                        s.pop(0)
                break
        else:
            raise TypeError("일관된 MRO를 만들 수 없습니다")

def compute_mro(cls, bases):
    if not bases:
        return [cls, object]
    # L[cls] = cls + merge(L[B1], L[B2], ..., [B1, B2, ...])
    # 여기서는 단순화 (bases에 mro가 이미 있다고 가정)
    seqs = [list(b.__mro__) for b in bases] + [list(bases)]
    return [cls] + c3_merge(seqs)

# 검증
class A: pass
class B(A): pass
class C(A): pass
class D(B, C): pass

computed = compute_mro(D, [B, C])
actual = list(D.__mro__)
print(computed)  # [D, B, C, A, object]
print(actual)    # [D, B, C, A, object]
print(computed == actual)  # True

용어 사전 (32개)

Dunder Method (던더 메서드)
__이름__ 형식의 메서드. Python 인터프리터가 특정 연산 시 자동으로 호출하는 약속된 메서드.
Magic Method (매직 메서드)
던더 메서드의 별칭. Special method라고도 불림.
__new__
객체 메모리 할당 담당. 새 인스턴스를 반환. 진짜 생성자. cls를 첫 인자로 받음.
__init__
객체 초기화 담당. 이미 만들어진 인스턴스(self)에 속성을 설정. 생성자가 아닌 초기화자.
__repr__
개발자용 문자열 표현. eval(repr(x)) == x가 이상적. 항상 구현해야 함.
__str__
사용자용 문자열 표현. print(), str()에서 사용. 없으면 __repr__ 폴백.
__format__
f-string 포맷 스펙 처리. f"{obj:spec}"에서 spec을 받음.
__eq__
== 연산자 구현. 오버라이드 시 __hash__가 자동으로 None이 됨.
__hash__
hash() 함수 구현. 딕셔너리 키, 세트에 필요. a==b → hash(a)==hash(b) 계약.
Identity (동일성)
같은 객체인지 — is 연산자. id()가 같으면 동일.
Equality (동등성)
값이 같은지 — == 연산자. __eq__로 정의.
NotImplemented
반사 메서드에서 반환하는 싱글톤. "이 타입은 처리 못함, 반대쪽에 물어봐".
Reflected Method (반사 메서드)
__radd__, __rmul__ 등. 왼쪽 피연산자가 NotImplemented 반환 시 오른쪽 피연산자가 시도.
MRO (Method Resolution Order)
메서드 탐색 순서. C3 선형화 알고리즘으로 결정됨. cls.mro() 또는 cls.__mro__로 확인.
C3 Linearization
Python 2.3+의 MRO 계산 알고리즘. 자식 우선, 부모 순서 유지, 일관성 검사.

MRO 상의 다음 클래스에 위임하는 프록시. "부모"가 아님. 다중 상속에서 안전한 호출.
Diamond Problem
A→B, A→C, D→B+C 구조에서 A의 메서드가 두 번 호출되는 문제. MRO+super()로 해결.
Mixin
단일 책임의 작은 클래스. 여러 믹스인을 상속해서 기능을 조합. 자체 인스턴스 생성 안 함.
Context Manager (컨텍스트 매니저)
__enter__와 __exit__을 구현한 객체. with문으로 자원을 안전하게 관리.
__enter__
with 블록 진입 시 호출. 반환값이 as 변수에 바인딩됨.
__exit__
with 블록 종료 시 항상 호출. exc_type, exc_val, exc_tb 3인자. True 반환 시 예외 억제.
contextmanager
contextlib의 데코레이터. yield 기반으로 컨텍스트 매니저를 간단히 만듦.
ExitStack
contextlib. 동적으로 컨텍스트 매니저를 쌓을 수 있는 도구. 개수가 런타임에 결정될 때.
try-except-else-finally
예외 처리 구조. else: 예외 없을 때만. finally: 항상. finally에서 return 금지.
raise from
예외 연쇄. raise NewError() from original_error. __cause__에 원인 저장.
BaseException
모든 예외의 최상위. SystemExit, KeyboardInterrupt를 포함. except Exception으로는 잡히지 않음.
total_ordering
functools 데코레이터. __eq__와 하나의 비교 메서드만으로 6개 비교 메서드를 자동 생성.
Singleton Pattern
인스턴스가 하나뿐인 클래스. __new__에서 구현. _instance 클래스 변수로 관리.
__init_subclass__
Python 3.6+. 서브클래스가 만들어질 때 자동 호출되는 훅. 플러그인 등록에 활용.
Hash Collision
다른 객체가 같은 해시값을 가지는 경우. 피할 수는 없지만 최소화해야 딕셔너리 성능 유지.
Hash Randomization
Python 3.3+. str/bytes 해시가 매 실행마다 다름. PYTHONHASHSEED로 고정 가능.
tp_slots
CPython 타입 객체의 C 함수 포인터 슬롯. 내장 연산자가 이 슬롯을 직접 참조 (인스턴스 딕셔너리 무시).

다음 주 맛보기 — 13주차: 데코레이터

클래스와 던더 메서드를 배웠으니, 이제 함수를 변환하는 패턴을 배울 차례입니다. 데코레이터는 함수를 인자로 받아 새 함수를 반환하는 패턴입니다. 

# 13주차 미리보기: 데코레이터
import time
from functools import wraps

def timer(func):
    @wraps(func)
    def wrapper(*args, **kwargs):
        start = time.perf_counter()
        result = func(*args, **kwargs)
        elapsed = time.perf_counter() - start
        print(f"{func.__name__}: {elapsed:.4f}초")
        return result
    return wrapper

@timer
def slow_sum(n):
    return sum(range(n))

slow_sum(1_000_000)   # slow_sum: 0.0XXX초
