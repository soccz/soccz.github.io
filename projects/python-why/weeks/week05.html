이번 주 이야기

ATM 앞에 서 있습니다. 카드를 넣고 10만 원을 요청했습니다. ATM은 0.1초 만에 "잔액이 부족합니다"를 출력합니다.

ATM은 어떻게 '판단'했을까요? 잔액(balance)과 요청액(request)을 비교하고, 조건에 따라 다른 동작을 했습니다. 이것이 조건문입니다. 

balance = 50000
request = 100000

if balance >= request:
    print("출금 완료")
    balance -= request
else:
    print("잔액이 부족합니다")

지난 주에 배운 비교 연산자(>=)와 Truthiness가 이번 주 조건문의 엔진입니다. if는 단순한 문법 기호가 아니라 — CPU의 조건 분기 명령(conditional branch instruction)을 Python 언어로 표현한 것입니다.

프로그램도 선택을 해야 한다

조건문이 없다면 프로그램은 항상 같은 일만 합니다. 어떤 상황에서도 "출금 완료"만 출력하거나, 항상 "잔액 부족"만 출력해야 하죠. 실생활의 모든 결정에는 조건이 있습니다:

  • 체력이 0이면 → 게임 오버
  • 비가 오면 → 우산을 가져간다
  • 성적이 90점 이상이면 → A학점

조건문은 프로그램이 상황에 따라 다르게 행동하게 만드는 도구입니다.

score = 85

if score >= 90:
    print("A")
elif score >= 80:
    print("B")
elif score >= 70:
    print("C")
else:
    print("F")

프로그램의 분기점(Branch)

조건문이 실행되는 시점에 프로그램은 두 갈래 이상의 길 앞에 섭니다. 조건이 참이면 한 쪽, 거짓이면 다른 쪽 — 이것을 분기(Branch)라고 합니다.

분기가 많을수록 프로그램은 복잡해집니다. 테스트해야 할 경우의 수도 늘어납니다. 조건문 하나 = 최소 2가지 경로. 조건문 두 개(독립) = 최대 4가지 경로. 

# 두 조건 → 최대 4가지 경로
is_logged_in = True
is_admin = False

if is_logged_in:
    if is_admin:
        print("관리자 대시보드")
    else:
        print("일반 사용자 페이지")
else:
    print("로그인 페이지")
실행 결과:
일반 사용자 페이지

구조적 프로그래밍 — 왜 if가 goto를 대체했는가

초기 프로그래밍 언어(어셈블리, FORTRAN 등)에는 GOTO라는 명령이 있었습니다. "줄 42로 점프하라"는 식이었죠. 코드가 복잡해질수록 실행 흐름을 추적하기 불가능해졌습니다. 이를 스파게티 코드라고 불렀습니다.

1968년 에츠허르 다익스트라(Dijkstra)는 "Go To Statement Considered Harmful"을 발표했습니다. 이후 if, for, while, def로 대표되는 구조적 프로그래밍(Structured Programming)이 표준이 되었습니다.

Python의 if/elif/else는 원서(reference/compound_stmts.txt)에서 "compound statement(복합문)"으로 분류됩니다. 단순한 표현식(expression)과 달리, 블록(block/suite)을 포함하는 문(statement)입니다.

Python 공식 문서(reference/compound_stmts §8.1):
"Compound statements contain (groups of) other statements; they affect or control the execution of those other statements in some way."

Part A — CPython의 ceval.c 조건 처리

Python의 if는 CPython bytecode의 POP_JUMP_IF_FALSE로 변환되고, 이는 ceval.c의 메인 루프에서 switch(opcode) 분기로 실행됩니다.

Python → CPython bytecode → C 인터프리터 → 기계어의 실행 흐름: 

Python: if n > 0:
    ↓
CPython bytecode: COMPARE_OP(>) → POP_JUMP_IF_FALSE(target)
    ↓
C: ceval.c의 switch(opcode) 분기
    ↓
x86-64: CMP rax, 0 / JLE target

Part B — CPU 분기 예측기(Branch Predictor)

현대 CPU는 분기 예측기(Branch Predictor)를 통해 조건이 평가되기 전에 어느 쪽 경로를 실행할지 미리 추측합니다. 예측 성공률이 높을수록 파이프라인 스톨 없이 빠르게 실행됩니다. "루프 안의 if" 패턴이 성능 민감 코드에서 중요한 이유입니다.

Suite, Clause, 그리고 표현식 vs 문

Python 원서(reference/compound_stmts)는 if 문을 정확하게 정의합니다: 

if_stmt ::=  "if" named_expr_test ":" suite
             ("elif" named_expr_test ":" suite)*
             ["else" ":" suite]

suite    ::=  stmt_list NEWLINE | NEWLINE INDENT statement+ DEDENT
  • named_expr_test: 조건 부분 — walrus 연산자(:=) 포함 가능
  • suite: if 블록의 본문 — 한 줄 또는 들여쓴 여러 줄
  • clause: if, elif, else 각각의 단위

핵심 구분: if는 문(statement)입니다. 표현식(expression)이 아닙니다. 문은 값을 반환하지 않습니다 — 따라서 result = if x > 0: ...은 문법 오류입니다. 값을 반환하는 "한 줄 if"는 result = "양수" if x > 0 else "음수" (조건식, §07 참조). 

# walrus operator (named_expr_test)와 if 조합
import re
text = "주문번호: ORD-12345"
if m := re.search(r"ORD-(\d+)", text):
    print(f"주문번호: {m.group(1)}")  # 매칭되면 실행, m을 바로 사용
실행 결과:
주문번호: 12345

Part A — POP_JUMP_IF_FALSE — if/elif/else의 바이트코드 구조

import dis

def classify(n):
    if n > 0:
        return "양수"
    elif n < 0:
        return "음수"
    else:
        return "영"

dis.dis(classify)
실행 결과 (Python 3.12):
4 0 RESUME 0 5 2 LOAD_FAST 0 (n) 4 LOAD_CONST 1 (0) 6 COMPARE_OP 68 (>) 10 POP_JUMP_IF_FALSE 1 (to 14) 6 12 RETURN_CONST 2 ('양수') 7 >> 14 LOAD_FAST 0 (n) 16 LOAD_CONST 1 (0) 18 COMPARE_OP 2 (<) 22 POP_JUMP_IF_FALSE 1 (to 26) 8 24 RETURN_CONST 3 ('음수') 10 >> 26 RETURN_CONST 4 ('영')

바이트코드 해독:

  • COMPARE_OP 68 (>): oparg 68 = (4<<4|8) — 상위 4비트가 비교 종류(4=GT), 하위 4비트가 캐시 힌트. n과 0을 비교, True/False를 스택에 push
  • POP_JUMP_IF_FALSE 1 (to 14): 스택 top이 False이면 offset 14로 점프 (elif 시작) → 스택 pop
  • True이면 점프하지 않고 RETURN_CONST '양수' 실행
  • offset 14(>> 표시 = 점프 목적지)부터 elif 절 시작
  • 모든 조건이 거짓이면 최종 RETURN_CONST '영'

COMPARE_OP가 4바이트인 이유: Python 3.12의 Specializing Adaptive Interpreter가 자주 호출되는 비교 연산을 타입별로 최적화(specialize)하기 위해 2바이트(1 cache entry)의 인라인 캐시를 추가한다. 초기에는 일반 경로로, 호출이 쌓이면 COMPARE_OP_INT 등으로 특화된다.

Part B — RESUME 명령어의 역할

모든 함수의 첫 줄에 등장하는 RESUME 0은 Python 3.11에서 도입됐다. 이 명령어는 세 가지를 처리한다:

  1. tracing/profiling hooks 체크 (sys.settrace 등)
  2. generator/coroutine의 throw() 처리
  3. 비동기 실행에서 send() 처리

일반 동기 함수에서는 거의 no-op이지만, 모든 함수 진입점에 일관성을 부여한다.

Part A (추가) — and/or 단락 평가의 바이트코드

위의 if/elif/else 바이트코드는 POP_JUMP_IF_FALSE로 분기했습니다. and/or는 다른 점프 명령어를 사용합니다:

import dis

# and/or 단락 평가의 바이트코드
def check(a, b):
    return a and b

def check_or(a, b):
    return a or b

print("=== and ===")
dis.dis(check)
print("=== or ===")
dis.dis(check_or)
핵심 바이트코드:
• COMPARE_OP: 비교 연산 (<, >, ==, !=, in, is)
• POP_JUMP_IF_FALSE: 거짓이면 점프 (if 조건 실패 시 분기)
• JUMP_IF_FALSE_OR_POP (Python 3.11 이전): and의 단락 평가 — 거짓이면 점프, 참이면 스택에서 제거
• JUMP_IF_TRUE_OR_POP (Python 3.11 이전): or의 단락 평가 — 참이면 점프, 거짓이면 스택에서 제거
※ Python 3.12+에서는 COPY + POP_JUMP_IF_FALSE/POP_JUMP_IF_TRUE로 대체됨

Part B (추가) — CPU 분기 예측과 조건문 최적화

현대 CPU는 분기 예측(Branch Prediction)을 통해 조건문 결과를 미리 추측하고 명령을 미리 실행합니다. 예측이 맞으면 속도 향상, 틀리면 파이프라인을 비워야 하므로 느려집니다 (분기 오예측 페널티). 

import timeit

# 정렬된 리스트: CPU가 패턴을 쉽게 예측
sorted_data = list(range(10000))
# 랜덤 리스트: CPU가 예측하기 어려움
import random
random_data = sorted_data.copy()
random.shuffle(random_data)

def count_large_sorted():
    count = 0
    for x in sorted_data:
        if x > 5000:   # 뒤쪽 절반에서만 True — 패턴 명확
            count += 1
    return count

def count_large_random():
    count = 0
    for x in random_data:
        if x > 5000:   # 랜덤 — 예측 어려움
            count += 1
    return count

t1 = timeit.timeit(count_large_sorted, number=1000)
t2 = timeit.timeit(count_large_random, number=1000)
print(f"정렬된 데이터: {t1:.3f}s")
print(f"랜덤 데이터:   {t2:.3f}s")
print(f"차이: {t2/t1:.1f}배")
Python에서의 시사점: CPython의 인터프리터 오버헤드가 크기 때문에 Python 레벨에서 분기 예측 효과는 미미합니다. 하지만 NumPy/Pandas/Cython 코드에서는 데이터 정렬이 성능에 영향을 줍니다. 가장 흔한 케이스를 if의 맨 앞에 배치하는 것이 가독성과 성능 모두에 좋습니다.

Truthiness를 조건으로 활용하기

초급

if result: 와 if result is not None: — 무슨 차이인가?

4주차 복습: 0, "", [], None, False는 Falsy. 나머지는 Truthy. 자세한 내용은 4주차 §06을 참고하세요.

조건문에서 가장 자주 발생하는 실수: Truthiness 체크와 None 체크를 혼동하는 것입니다. 

# 상황: 함수가 실패하면 None, 성공하면 값을 반환한다
def find_age(name):
    db = {"민지": 30, "수호": 0}   # 수호의 나이는 0살
    return db.get(name)            # 없으면 None 반환

age = find_age("수호")
print(age)   # 0

# 잘못된 체크: 0도 Falsy이므로 "나이 없음" 출력!
if age:
    print(f"나이: {age}")
else:
    print("나이 없음")   # ← 버그! 수호의 나이는 0인데 없다고 함

# 올바른 체크: None만 걸러내기
if age is not None:
    print(f"나이: {age}")   # 나이: 0
else:
    print("나이 없음")
실행 결과:
0 나이 없음 나이: 0

세 가지 조건 체크 방식 — 언제 무엇을 쓰는가

같은 변수에 대해 세 가지 방식이 있습니다. 명확한 선택 기준을 알아야 합니다:

def process(value):
    # 방식 1: if value: — "유효한 값이 있는가?"
    # 0, "", [], {}, None, False 모두 "없는 것"으로 취급
    if value:
        print(f"[1] 처리: {value}")

    # 방식 2: if value is not None: — "None이 아닌가?"
    # 0, "", []는 통과 → 함수가 성공은 했지만 결과가 0/빈 값인 경우에 사용
    if value is not None:
        print(f"[2] None 아님: {value}")

커스텀 클래스에서 Truthiness 설계하기

__bool__과 __len__을 정의하면 자신의 클래스가 if 조건에서 어떻게 동작할지 제어할 수 있습니다. 설계 원칙: "이 객체가 의미 있는 상태인가?"를 나타내도록 정의하세요. 

class Account:
    def __init__(self, balance):
        self.balance = balance

    def __bool__(self):
        return self.balance > 0   # 잔액 양수 = 사용 가능 상태

class Stack:
    def __init__(self):
        self.items = []
    def push(self, x):
        self.items.append(x)
    def __len__(self):
        return len(self.items)    # __bool__ 없으면 __len__으로 대체

# 이제 if 조건에서 자연스럽게 사용 가능
accounts = [Account(1000), Account(0), Account(-50)]
for acc in accounts:
    if acc:
        print(f"잔액 {acc.balance}: 사용 가능")
    else:
        print(f"잔액 {acc.balance}: 사용 불가")

s = Stack()
print(f"스택 비어있음: {not s}")  # __len__ 0 → Falsy
s.push(1)
print(f"스택 비어있음: {not s}")  # __len__ 1 → Truthy
실행 결과:

설계 함정: __bool__을 정의하면 __len__은 Truthiness에 영향을 주지 않습니다. 둘 다 정의했다면 __bool__이 항상 우선합니다.

Part A — nb_bool의 반환값 -1과 예외 전파

4주차 §06에서 PyObject_IsTrue()의 C 구현을 봤습니다. 이번에는 __bool__이 예외를 raise할 때 어떤 일이 벌어지는지 살펴봅니다: 

class Broken:
    def __bool__(self):
        raise ValueError("Truthiness를 알 수 없습니다")

obj = Broken()
try:
    if obj:          # PyObject_IsTrue 호출 → nb_bool → 예외 발생
        print("True")
except ValueError as e:
    print(f"예외: {e}")   # 예외: Truthiness를 알 수 없습니다
실행 결과:
예외: Truthiness를 알 수 없습니다

CPython 내부에서 일어나는 일:

// Objects/object.c — PyObject_IsTrue()
res = (*Py_TYPE(v)->tp_as_number->nb_bool)(v);
// __bool__이 예외를 raise하면 nb_bool은 -1을 반환
// (Python C API 규약: 예외 발생 시 정수 반환 함수는 -1)
return (res > 0) ? 1 : (res == 0 ? 0 : -1);
//                                    ↑ -1 반환 = "예외가 설정되어 있음"

// ceval.c — POP_JUMP_IF_FALSE 처리
int err = PyObject_IsTrue(cond);
if (err > 0)  { /* True 분기 */ }
else if (err == 0) { /* False 분기, jump */ }
else { goto error; }   // err == -1: 예외 전파

이것이 Python의 에러 처리 모델의 핵심: C 함수에서 -1은 "예외가 설정됐으니 처리를 중단하라"는 신호다. goto error는 CPython의 예외 전파 메커니즘으로 이어진다.

    print(f"안녕하세요, {display}님!")

# not (A or B)  == (not A) and (not B)

        return "둘 다 True"

5 0 RESUME 0 6 2 LOAD_FAST 0 (a) 4 POP_JUMP_IF_FALSE 3 (to 12) 6 LOAD_FAST 1 (b) 8 POP_JUMP_IF_FALSE 1 (to 12) 7 10 RETURN_CONST 1 ('둘 다 True') 8 >> 12 RETURN_CONST 2 ('아님')

관찰 포인트:

  • a가 Falsy이면 첫 번째 POP_JUMP_IF_FALSE가 offset 12로 점프 — b는 평가조차 안 됨

  • Python 3.11 이전의 JUMP_IF_FALSE_OR_POP / JUMP_IF_TRUE_OR_POP은 3.12에서 제거됨 — POP_JUMP_IF_FALSE / POP_JUMP_IF_TRUE로 통일

Guard Clause — 조기 반환으로 중첩 제거

Guard Clause(또는 Early Return)는 실패 조건을 먼저 처리하고 함수를 빠져나와 중첩을 줄이는 패턴입니다: 

# Before: 깊은 중첩
def play_content(is_logged_in, is_premium, has_content):
    if is_logged_in:
        if is_premium:
            if has_content:
                return "콘텐츠 재생"
            else:
                return "콘텐츠 없음"
        else:
            return "프리미엄 필요"
    else:
        return "로그인 필요"

# After: Guard Clause (조기 반환)
def play_content_v2(is_logged_in, is_premium, has_content):
    if not is_logged_in:
        return "로그인 필요"      # 실패 조건 먼저
    if not is_premium:
        return "프리미엄 필요"
    if not has_content:
        return "콘텐츠 없음"
    return "콘텐츠 재생"          # 성공은 마지막

print(play_content_v2(True, True, True))   # 콘텐츠 재생
print(play_content_v2(False, True, True))  # 로그인 필요
실행 결과:
콘텐츠 재생 로그인 필요
고급

순환 복잡도(Cyclomatic Complexity)와 코드 품질

Cyclomatic Complexity는 코드의 독립적인 실행 경로 수를 측정합니다. if 분기 하나 = +1, elif = +1, else = +0. 

# Cyclomatic Complexity가 높은 함수 (CC = 6)
def process(x, y, flag):
    result = 0
    if x > 0:              # +1
        if y > 0:          # +1
            result = x * y
        elif y < 0:        # +1
            result = x + y
    elif x < 0:            # +1
        if flag:           # +1
            result = -x
    return result

# 리팩토링: Guard Clause + 조건 분리 (CC = 4, flag 매개변수 제거로 로직 단순화)
def process_v2(x, y):
    if x <= 0:
        return 0
    if y > 0:
        return x * y
    if y < 0:
        return x + y
    return x

Python 코드 품질 도구인 radon이나 flake8은 Cyclomatic Complexity가 10을 넘으면 경고를 줍니다. Guard Clause는 CC를 낮추면서 가독성도 높이는 실용적인 패턴입니다.

현대 CPU는 분기 예측기(Branch Predictor)를 사용해 조건 분기의 결과를 미리 추측하고 파이프라인을 채웁니다. 예측이 틀리면 파이프라인 플러시(flush)가 발생해 성능이 저하됩니다. 

import timeit

# 예측하기 쉬운 분기: 패턴이 단조로움 (성능 유리)
def predictable(n):
    result = 0
    for i in range(n):
        if i < n // 2:      # 처음엔 항상 True, 나중엔 항상 False
            result += i
    return result

# 예측하기 어려운 분기: 무작위 패턴
import random
data = [random.choice([True, False]) for _ in range(10000)]
def unpredictable():
    result = 0
    for flag in data:
        if flag:
            result += 1
    return result

# 동일한 연산이지만 분기 예측 가능 여부에 따라 수십%까지 속도 차이 발생
# Python 레벨에서는 GIL과 인터프리터 오버헤드가 더 크지만,
# 성능 민감 코드는 이를 고려해 조건 순서를 정렬한다.

Python에서는 CPython 인터프리터 오버헤드가 분기 예측 효과를 희석하지만, NumPy/Cython/C extension을 활용할 때는 분기 패턴이 중요해진다. 실용 팁: 루프 안의 조건문에서 "더 자주 True인 조건"을 앞에 배치하면 단락 평가와 분기 예측 모두에서 이득을 얻는다.

표현식(Expression)과 문(Statement)의 차이

Python 원서(reference/expressions)에서 조건식은 "conditional expression"으로 정의됩니다: 

conditional_expression ::=  or_test ["if" or_test "else" expression]

이것이 표현식이기 때문에:

  • 값을 반환합니다 → 변수에 대입, 함수 인자, 리스트 안에 사용 가능
  • 타입이 있습니다 → type("양수" if True else 0) = str

반면 if ... else ... 블록(compound statement)은 값을 반환하지 않습니다. 이는 Java/C의 삼항 연산자 condition ? a : b와 기능적으로 동일하지만, Python은 더 영어 문장에 가까운 순서(a if condition else b)를 택했습니다.

numbers = [-3, -1, 0, 2, 5]

if 블록과의 차이:

  • if 블록: True 분기에서 바로 RETURN_CONST 가능 (함수 종료)
  • 조건식: True 분기 실행 후 JUMP_FORWARD로 False 분기를 건너뜀
  • 두 분기가 같은 STORE_FAST(offset 20)로 합류

return 문맥에서는 JUMP_FORWARD 없이 각 분기에서 바로 RETURN_CONST가 가능해 더 간결한 바이트코드가 생성된다:

    return "양수" if n > 0 else "음수나 영"

from dataclasses import dataclass

@dataclass
class Point:
    x: float
    y: float

@dataclass
class Circle:
    center: Point
    radius: float

@dataclass
class Rectangle:
    top_left: Point
    bottom_right: Point

def describe(shape):
    match shape:
        case Point(x=0, y=0):
            return "원점"
        case Point(x=x, y=0):
            return f"x축 위 ({x}, 0)"
        case Circle(center=Point(x=0, y=0), radius=r):
            return f"원점 중심 원, 반지름={r}"
        case Circle(center=c, radius=r):
            return f"원: 중심=({c.x}, {c.y}), 반지름={r}"
        case Rectangle(top_left=Point(x=x1, y=y1),
                       bottom_right=Point(x=x2, y=y2)):
            return f"사각형: ({x1},{y1})-({x2},{y2})"
        case _:
            return "알 수 없는 도형"

print(describe(Point(0, 0)))                          # 원점
print(describe(Circle(Point(0, 0), 5)))               # 원점 중심 원, 반지름=5
print(describe(Rectangle(Point(0, 3), Point(4, 0)))) # 사각형: (0,3)-(4,0)

Part A — match/case의 바이트코드 전략

match/case는 패턴 유형에 따라 다른 바이트코드를 사용한다: 

패턴 유형          → 바이트코드
─────────────────────────────────────
리터럴 패턴         → COPY + COMPARE_OP(==) + POP_JUMP_IF_FALSE
OR 패턴 (|)        → 각 패턴을 순서대로 시도
캡처 변수 (_이 아닌) → STORE_FAST (항상 매칭)
와일드카드 (_)      → 바이트코드 없음 (무조건 매칭)
시퀀스 패턴         → MATCH_SEQUENCE + GET_LEN + 길이 비교
매핑 패턴           → MATCH_MAPPING + MATCH_KEYS
클래스 패턴         → MATCH_CLASS

Part B — MATCH_CLASS의 동작 원리

import dis
from dataclasses import dataclass

@dataclass
class Point:
    x: float
    y: float

def check(p):
    match p:
        case Point(x=0, y=y_val):
            return y_val
        case _:
            return None

dis.dis(check)
핵심 바이트코드 (Python 3.12, 간략화):
2 LOAD_FAST 0 (p) MATCH_CLASS Point 0 ('x', 'y') # 실패하면 POP_JUMP_IF_NONE → 다음 case로 LOAD_CONST 0 # x == 0 비교 COMPARE_OP 2 (==) POP_JUMP_IF_FALSE # x != 0이면 다음 case로 STORE_FAST # y_val에 캡처된 y값 저장

MATCH_CLASS는 내부적으로:

  1. isinstance(subject, Point) 체크 (실패 시 None 반환)
  2. 위치 인자 패턴이 있으면 __match_args__ 속성으로 키 매핑
  3. 키워드 패턴은 getattr(subject, key)로 각 속성 추출
  4. 추출된 값들을 스택에 push → 개별 값 비교나 캡처로 이어짐
Python 공식 문서(reference/compound_stmts §8.6 — match):
"A match statement takes an expression and compares its value to successive patterns given as one or more case blocks." 클래스 패턴은 __match_args__를 통해 위치 인자를 지원한다 (PEP 634 §Class Patterns).

CPython 컴파일러는 조건문을 바이트코드로 변환할 때, 각 분기마다 POP_JUMP_IF_FALSE / POP_JUMP_IF_TRUE 명령을 삽입합니다. AI가 생성한 조건문 코드에서 경계 조건(edge case)이 누락되면, 특정 분기 경로가 테스트되지 않은 채 남게 됩니다. 

import dis

def grade(score):
    """점수를 학점으로 변환. 유효 범위: 0-100"""
    if score < 0 or score > 100:
        raise ValueError(f"유효하지 않은 점수: {score}")
    if score >= 90: return "A"
    if score >= 80: return "B"
    if score >= 70: return "C"
    if score >= 60: return "D"
    return "F"

dis.dis(grade)
# 각 if 문마다 COMPARE_OP + POP_JUMP_IF_FALSE 쌍이 생성됨
# 첫 번째 if의 or는 두 개의 COMPARE_OP으로 분해:
#   score < 0 → POP_JUMP_IF_TRUE (단락평가: True면 바로 raise)
#   score > 100 → POP_JUMP_IF_FALSE (False면 다음 if로)

Part B — 분기 커버리지와 테스트 전략 (소프트웨어 테스팅 이론)

조건문이 많은 코드는 분기 누락이 실무 버그의 주요 원인입니다. 구조적 테스트 기법으로 모든 분기를 커버해야 합니다. 

# 분기 커버리지 100%를 위한 테스트 케이스
import unittest

class TestGrade(unittest.TestCase):
    # 경계값 분석 (Boundary Value Analysis)
    def test_a_boundary(self):
        self.assertEqual(grade(90), "A")   # 경계값
        self.assertEqual(grade(100), "A")  # 최대값
    def test_b_boundary(self):
        self.assertEqual(grade(89), "B")   # A 경계 바로 아래
        self.assertEqual(grade(80), "B")
    def test_f(self):
        self.assertEqual(grade(0), "F")    # 최소값
        self.assertEqual(grade(59), "F")   # D 경계 바로 아래
    def test_invalid(self):
        with self.assertRaises(ValueError):
            grade(-1)
        with self.assertRaises(ValueError):
            grade(101)

# 각 분기(if, elif, else)마다 최소 하나의 테스트 케이스
# 경계값(boundary): 조건의 경계에서 테스트 (89/90, 79/80, 69/70...)

Branch Coverage: 모든 분기의 참/거짓 경로를 테스트했는가?
MC/DC (Modified Condition/Decision Coverage): 항공/의료 소프트웨어 수준 — 각 조건이 독립적으로 결과에 영향을 주는지 확인. score < 0 or score > 100에서 MC/DC를 만족하려면, 각 조건이 독립적으로 결과를 바꾸는 테스트 쌍이 필요합니다 (예: score=-1 vs score=50은 좌측 조건만 변경, score=50 vs score=101은 우측 조건만 변경).

용어 사전 (22개)

Conditional Statement (조건문)

조건의 참/거짓에 따라 다른 코드 블록을 실행하는 제어 구조. Python의 if/elif/else.

Branch (분기)

조건문에 의해 생성되는 실행 경로의 갈림길. 조건이 참이면 한 쪽, 거짓이면 다른 쪽.

Compound Statement (복합문)

다른 문(statement)을 포함하는 문. if, for, def, class 등.

Suite (블록)

Python 원서 용어. 들여쓰기로 구분된 문의 묶음. if/for/def 등의 본문.

Clause (절)

if 문의 구성 요소. if 절, elif 절, else 절.

Truthiness (참같음 — 개념)

불리언 컨텍스트에서 참/거짓을 판단하는 개념적 속성. __bool__ → __len__ 순으로 판단하는 프로토콜 자체를 가리킨다.

Falsy (거짓같음)

불리언 컨텍스트에서 False로 평가되는 값. False, 0, "", [], None 등.

Truthy (참같은 값)

bool 컨텍스트에서 True로 평가되는 개별 값. Falsy가 아닌 모든 객체. Truthiness는 이 판단 체계 자체를, Truthy는 그 판단 결과 참인 개별 값을 가리킨다.

Short-Circuit Evaluation (단락 평가)

and/or 연산에서 결과가 확정되면 나머지 피연산자 평가를 중단하는 동작.

Guard Clause (경비 조항)

함수 시작 부분에서 실패 조건을 먼저 처리하고 early return해 중첩을 줄이는 패턴.

Early Return (조기 반환)

함수 끝이 아닌 중간에 return을 사용해 Guard Clause를 구현하는 기법.

Conditional Expression (조건식)

A if 조건 else B 형태의 표현식. 값을 반환하므로 대입/인자/리스트 안에 사용 가능.

Ternary Operator (삼항 연산자)

세 피연산자를 갖는 연산자. Python의 조건식 A if C else B가 해당.

Structural Pattern Matching (구조적 패턴 매칭)

Python 3.10 PEP 634. match/case로 값의 구조(타입, 속성, 시퀀스)를 동시에 매칭.

Wildcard Pattern (와일드카드 패턴)

match/case에서 case _:. 어떤 값도 매칭. 변수로 캡처하지 않음.

Capture Pattern (캡처 패턴)

match/case에서 case x:처럼 변수명으로 값을 캡처하는 패턴.

OR Pattern (OR 패턴)

match/case에서 case A | B | C:처럼 여러 패턴 중 하나와 매칭.

Class Pattern (클래스 패턴)

match/case에서 case Point(x=0):처럼 타입과 속성을 동시에 매칭.

POP_JUMP_IF_FALSE

CPython 바이트코드. 스택 top을 pop하고, False이면 목적 주소로 점프. if의 핵심 명령어.

MATCH_CLASS

CPython 바이트코드 (Python 3.10+). isinstance 체크 + 속성 추출. 클래스 패턴 구현.

Cyclomatic Complexity (순환 복잡도)

코드의 독립적 실행 경로 수. 분기(if, elif, while 등) 하나당 +1. 10 이하 권장.

다음 주 맛보기 — 리스트와 튜플

지금까지 우리는 하나의 값(x = 42)만 다뤘습니다. 6주차에서는 여러 값을 담는 컨테이너를 배웁니다: 

# 리스트: 순서 있고, 변경 가능
scores = [95, 82, 78, 91, 65]

# 조건문과 결합
for score in scores:        # 다음 주에!
    if score >= 90:
        print(f"{score}: A")
    elif score >= 80:
        print(f"{score}: B")
    else:
        print(f"{score}: C이하")

핵심 질문: scores[0]은 어떻게 동작하는가? 리스트가 변경 가능(Mutable)이고 튜플이 변경 불가(Immutable)인 이유는? 그리고 리스트는 컴퓨터 메모리에서 어떻게 생겼는가?
