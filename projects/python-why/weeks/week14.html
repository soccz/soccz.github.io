01 — 이번 주 이야기

지난 주에 데코레이터를 배웠습니다. 함수를 감싸서 기능을 추가하는 방법. 그런데 프로젝트가 커질수록 다른 문제가 생깁니다: 

# 이 함수가 뭘 받고 뭘 반환하는지 알 수 있나요?
def process(data, config, mode):
    ...
    return result

세 달 후 이 코드를 다시 보면: data가 리스트인가, 딕셔너리인가? config는 어떤 키를 가져야 하나? result는 무슨 타입인가? 

# 타입 힌트를 붙이면 — 코드 자체가 문서가 됩니다
def process(
    data: list[dict[str, int]],
    config: dict[str, str],
    mode: str
) -> list[int]:
    ...
    return result

함수 시그니처만으로 의도가 보입니다. 하지만 Python은 이것을 강제하지 않습니다 — "힌트(hint)"입니다.

오늘은 이 힌트가 어떻게 저장되고, 도구들이 어떻게 활용하며, 구조적 서브타이핑(Protocol)이라는 강력한 개념까지 완전히 분해합니다.

02 — 타입 힌트는 왜 "힌트"인가

핵심 비유: 요리 레시피의 "추천 재료"

요리책에 "달걀 2개 권장"이라고 적혀 있습니다. 메추리알 4개를 쓰거나 달걀을 빼도 요리책이 막지는 않습니다.

Python 타입 힌트는 이 추천 재료 설명서입니다 — 어기는 것을 막지 않고, 다른 사람(또는 mypy 같은 도구)에게 의도를 알려줍니다.

초급

먼저, 불편함을 느껴봅시다

동료가 만든 함수를 사용해야 합니다:

# 동료가 만든 함수
def process(data, config, mode):
    ...

# 질문: data에 뭘 넣어야 하나요? 리스트? 딕셔너리? 문자열?
# config는? mode는 숫자? 문자열?
# 반환값은 뭔가요?

# 방법 1: 소스코드를 전부 읽는다 (시간 소모)
# 방법 2: 직접 실행해보고 에러를 본다 (비효율)
# 방법 3: 독스트링을 읽는다 (있으면...)

# 타입 힌트가 있으면:
def process(
    data: list[dict[str, int]],
    config: dict[str, str],
    mode: str
) -> list[int]:
    ...
# 함수 시그니처만 보면 됩니다. 소스를 읽을 필요가 없습니다.

타입 힌트 ≠ 타입 검사

def greet(name: str) -> str:
    return f"안녕, {name}"

# str을 기대하지만 int를 넘겨도 실행됩니다!
print(greet(42))   # 안녕, 42

# Python은 타입 힌트를 완전히 무시합니다
# "힌트"일 뿐이지 "규칙"이 아닙니다
가장 흔한 오해: "타입 힌트를 쓰면 타입이 강제된다" → 아닙니다. Python 인터프리터는 타입 힌트를 완전히 무시합니다. 타입을 검사하는 것은 mypy 같은 별도의 도구입니다.
그러면 왜 쓰는 건가요?
세 가지 이유:
  1. 문서화: 함수가 무엇을 기대하는지 한눈에 보입니다
  2. IDE 지원: 자동완성, 호버 정보, 인라인 에러 표시
  3. 버그 사전 탐지: mypy 같은 정적 분석 도구가 실행 전에 타입 오류를 찾습니다
중급

__annotations__ — 힌트는 어디에 저장되는가

def add(a: int, b: int) -> int:
    return a + b

# 타입 힌트는 __annotations__ 딕셔너리에 저장됩니다
print(add.__annotations__)
# {'a': <class 'int'>, 'b': <class 'int'>, 'return': <class 'int'>}

# 클래스 변수도 마찬가지
class User:
    name: str
    age: int

print(User.__annotations__)
# {'name': <class 'str'>, 'age': <class 'int'>}
Python은 타입 힌트를 __annotations__ 딕셔너리에 저장만 하고 아무 동작도 하지 않습니다. mypy, IDE, 프레임워크가 이 딕셔너리를 읽어서 활용합니다.

변수 타입 힌트

# 변수에도 타입 힌트를 붙일 수 있습니다 (PEP 526)
name: str = "철수"
age: int = 25
scores: list[int] = [90, 85, 100]

# 초기값 없이 선언만도 가능 (실제로 변수가 생성되지는 않음)
email: str      # __annotations__에만 기록됨

import sys
print(sys.modules[__name__].__annotations__)
# {'name': str, 'age': int, 'scores': list[int], 'email': str}
고급

PEP 3107 → PEP 484 → PEP 526 → PEP 604: 타입 힌트의 역사

타입 힌트의 진화:
  • PEP 3107 (Python 3.0, 2006): 함수 어노테이션 문법 도입. 의미는 정의 안 함
  • PEP 484 (Python 3.5, 2015): typing 모듈 + 타입 힌트 표준화
  • PEP 526 (Python 3.6, 2016): 변수 어노테이션 문법
  • PEP 585 (Python 3.9, 2020): list[int] 소문자 내장 타입 직접 사용
  • PEP 604 (Python 3.10, 2021): int | str Union 단축 문법
  • PEP 695 (Python 3.12, 2023): type X = ... 타입 별칭 문법
전공자

Part A: __annotations__의 저장 메커니즘 — CPython 내부

Part A: CPython에서 함수의 __annotations__는 PyFunctionObject.func_annotations에 저장됩니다. def f(x: int): 컴파일 시 int가 평가되어 dict에 저장됩니다. Python 3.14+에서는 PEP 749로 어노테이션이 지연 평가(lazy evaluation)됩니다 — 어노테이션 표현식은 함수 정의 시점에 실행되지 않고, get_type_hints() 호출 시 비로소 평가됩니다. 이전까지는 from __future__ import annotations(PEP 563)로 문자열화하여 지연을 시뮬레이션했습니다.

Part B: 정적 타입 vs 동적 타입 — 타입 이론

정적 타입(Static typing): 변수가 선언 시 타입을 가짐 (Java, C, Rust). 컴파일 타임에 모든 타입 오류를 잡을 수 있음.
동적 타입(Dynamic typing): 변수가 아닌 값이 타입을 가짐 (Python, JavaScript, Ruby). 런타임에만 타입이 결정됨.
점진적 타입(Gradual typing): Python의 접근법. 타입 힌트가 있는 부분만 정적 검사, 나머지는 동적으로 동작. Any 타입이 정적/동적 경계를 연결합니다.

이 이론은 Jeremy Siek의 2006년 논문 "Gradual Typing for Functional Languages"에서 체계화되었고, Guido van Rossum이 PEP 484를 설계할 때 직접 참조했습니다.

03 — 타입 힌트 문법 총정리

핵심 비유: 주소 체계

타입 힌트는 코드의 우편번호입니다. 없어도 편지가 도착하지만, 있으면 배달원(IDE, mypy)이 훨씬 빠르고 정확하게 일합니다. 잘못된 우편번호가 있으면 배달원이 "주소가 이상해요!"라고 알려줍니다.

초급

기본 타입 힌트

# 함수 매개변수와 반환값
def greet(name: str) -> str:
    return f"안녕, {name}!"

def add(a: int, b: int) -> int:
    return a + b

def average(numbers: list) -> float:
    return sum(numbers) / len(numbers)

# 변수
name: str = "철수"
age: int = 25
is_student: bool = True
height: float = 175.5
->는 반환 타입을 나타냅니다. def greet(name: str) -> str:는 "str을 받아 str을 반환하는 함수"를 의미합니다.
반환값이 없는 함수는 어떻게 표기하나요?
None을 사용합니다: 
def print_hello() -> None:
    print("안녕!")

# None은 "반환값 없음"을 명시합니다
# return 문이 없거나 return만 있는 경우
중급

컬렉션 타입 힌트

# Python 3.9+: 내장 타입을 직접 사용 (PEP 585)
scores: list[int] = [90, 85, 100]
names: tuple[str, ...] = ("철수", "영희", "민수")    # 가변 길이 튜플
pair: tuple[str, int] = ("철수", 25)                 # 고정 길이 튜플

Optional과 Union

from typing import Optional, Union

# Optional[X] = X | None
def find_user(user_id: int) -> Optional[str]:
    """유저를 찾으면 이름 반환, 없으면 None"""
    if user_id == 1:
        return "철수"
    return None

# Union[X, Y] = X 또는 Y
def process(data: Union[str, list[str]]) -> list[str]:
    if isinstance(data, str):
        return [data]
    return data

# Python 3.10+: | 연산자로 Union 대체 (PEP 604)
def process_new(data: str | list[str]) -> list[str]:
    if isinstance(data, str):
        return [data]
    return data
Optional[str]은 "선택적 매개변수"가 아닙니다. Optional[str]은 str | None과 동일합니다. "값이 str이거나 None일 수 있다"는 뜻이지, 매개변수를 생략 가능하다는 뜻이 아닙니다.
Optional 매개변수를 만들려면?
기본값을 지정하면 됩니다: 
# Optional "매개변수" = 기본값 지정
def greet(name: str, greeting: str = "안녕") -> str:
    return f"{greeting}, {name}!"

greet("철수")             # greeting 생략 가능
greet("철수", "Hello")    # greeting 지정 가능

# Optional "타입" = None이 될 수 있는 값
def find(items: list[int], target: int) -> int | None:
    try: return items.index(target)
    except ValueError: return None
고급

Any와 타입 좁히기 (Type Narrowing)

from typing import Any

# Any = "아무 타입이든 OK" — 타입 검사를 우회
def log(message: Any) -> None:
    print(f"[LOG] {message}")

# Any는 모든 타입과 호환됩니다
log("문자열")
log(42)
log([1, 2, 3])

# isinstance로 타입 좁히기 (Type Guard)
def double(x: int | str) -> int | str:
    if isinstance(x, int):
        return x * 2       # mypy는 여기서 x를 int로 인식
    else:
        return x + x       # mypy는 여기서 x를 str로 인식

print(double(5))       # 10
print(double("abc"))   # abcabc
Any 남용 주의: Any를 쓰면 해당 부분의 타입 검사가 완전히 비활성화됩니다. "타입을 모르겠으니 일단 Any"로 시작하면 타입 힌트의 의미가 없어집니다. object를 사용하면 "모든 타입을 허용하되 타입 안전성은 유지"됩니다.
전공자

Part A: PEP 585 — __class_getitem__의 마법

Part A: list[int]가 동작하려면 list.__class_getitem__(int)가 호출되어야 합니다. Python 3.9에서 PEP 585에 의해 list, dict 등 내장 타입에 __class_getitem__이 직접 구현되어 (PEP 560으로 메커니즘 자체는 3.7에서 도입) C 구현 내장 타입(list, dict, tuple, set)도 제네릭 문법을 지원합니다. 반환값은 types.GenericAlias 객체입니다: 
import types
alias = list[int]
print(type(alias))          # <class 'types.GenericAlias'>
print(alias.__origin__)     # <class 'list'>
print(alias.__args__)       # (<class 'int'>,)

# 런타임에서는 list[int]()와 list()가 동일합니다
print(list[int]() == [])    # True

Part B: 왜 Variance가 중요한가 — list[Dog]는 list[Animal]이 아니다

직관적으로 "Dog가 Animal의 하위 타입이면, Dog 리스트도 Animal 리스트 아닌가?"라고 생각합니다. 하지만 이것은 위험합니다: 

# 만약 list[Dog]가 list[Animal]의 서브타입이라면...
class Animal: pass
class Dog(Animal): pass
class Cat(Animal): pass

dogs: list[Dog] = [Dog(), Dog()]

# 이 함수는 list[Animal]을 받으니 list[Dog]도 받을 수 있다?
def add_cat(animals: list[Animal]) -> None:
    animals.append(Cat())   # Cat을 추가!

add_cat(dogs)   # dogs에 Cat이 들어감 — 타입 안전성 파괴!

이것이 Mutable 컨테이너가 Invariant(무공변)인 이유입니다. 쓰기가 가능하면 "하위 타입의 컨테이너를 상위 타입으로 취급"하는 순간 잘못된 타입이 삽입될 수 있습니다.

읽기만 가능한 컨테이너는 안전합니다(Covariant): Sequence[Dog]에서 꺼내면 Dog이고, Dog는 Animal이므로 Sequence[Animal]로 취급해도 문제없습니다 — 추가할 수 없으니까.

함수 매개변수는 반대(Contravariant): "모든 Animal을 처리하는 함수"는 당연히 Dog도 처리할 수 있으므로 Callable[[Animal], None]을 Callable[[Dog], None] 대신 쓸 수 있습니다. 이 규칙은 Barbara Liskov의 치환 원칙(LSP, 1987)에서 비롯됩니다.

요약: 읽기만 → 공변(Covariant), 쓰기 가능 → 불변(Invariant), 매개변수 → 반변(Contravariant). Java는 이것을 ? extends T/? super T로, Python은 TypeVar(covariant=True)/TypeVar(contravariant=True)로 표현합니다.

04 — TypeVar & Generic: 타입의 변수

핵심 비유: 택배 상자의 라벨

택배 상자에 "내용물: ___"이라는 빈 라벨이 있습니다. 과일을 넣으면 "과일 상자"가 되고, 책을 넣으면 "책 상자"가 됩니다. 상자 구조는 같지만 내용물 타입이 다른 것 — 이것이 Generic입니다. TypeVar는 그 빈 라벨(타입 변수)입니다.

중급

TypeVar — 타입을 변수처럼

from typing import TypeVar

T = TypeVar('T')

def first(items: list[T]) -> T:
    """리스트의 첫 번째 요소를 반환"""
    return items[0]

# T가 int로 결정됨
x: int = first([1, 2, 3])       # T = int → 반환 int

# T가 str로 결정됨
y: str = first(["a", "b"])      # T = str → 반환 str

# T가 float로 결정됨
z: float = first([1.1, 2.2])    # T = float → 반환 float
TypeVar는 "같은 타입"이 여러 곳에 나타나야 할 때 사용합니다. "리스트의 요소 타입 = 반환 타입"이라는 관계를 표현합니다.

TypeVar with bound

from typing import TypeVar

# bound: T는 이 타입 또는 그 서브타입만 허용
from typing import SupportsFloat
T = TypeVar('T', bound=SupportsFloat)

def max_of(a: T, b: T) -> T:
    return a if a > b else b

print(max_of(3, 7))        # 7  ✓ (int는 SupportsFloat 구현)
print(max_of(3.5, 2.1))    # 3.5  ✓

# TypeVar with constraints
Number = TypeVar('Number', int, float)   # int 또는 float만 허용

def add(a: Number, b: Number) -> Number:
    return a + b
고급

Generic 클래스 — 자신만의 제네릭 만들기

from typing import TypeVar, Generic

T = TypeVar('T')

class Stack(Generic[T]):
    def __init__(self) -> None:
        self._items: list[T] = []

    def push(self, item: T) -> None:
        self._items.append(item)

    def pop(self) -> T:
        return self._items.pop()

    def peek(self) -> T:
        return self._items[-1]

    def __len__(self) -> int:
        return len(self._items)

    def __repr__(self) -> str:
        return f"Stack({self._items})"

# 사용 (mypy가 타입을 추적)
int_stack: Stack[int] = Stack()
int_stack.push(1)
int_stack.push(2)
# int_stack.push("str")   # mypy 에러! Stack[int]에 str 넣기
val: int = int_stack.pop()  # mypy가 int로 추론

str_stack: Stack[str] = Stack()
str_stack.push("hello")
# 런타임에는 아무 제약 없음 — mypy만 검사
Python 3.12+에서는 더 간단합니다 
# PEP 695 (Python 3.12+): 새로운 타입 매개변수 문법
class Stack[T]:
    def __init__(self) -> None:
        self._items: list[T] = []

    def push(self, item: T) -> None:
        self._items.append(item)

    def pop(self) -> T:
        return self._items.pop()

def first[T](items: list[T]) -> T:
    return items[0]

Part B: Parametric Polymorphism — 타입 이론

Generic은 타입 이론의 매개변수적 다형성(Parametric Polymorphism)입니다. ML(1973), Haskell, Java Generics, C++ Templates가 모두 이 개념의 구현입니다. System F(Lambda Calculus + 타입 추상화)에서 유래했으며, "타입에 대해 추상화된 함수"를 정의할 수 있게 합니다.

Python의 TypeVar는 Java의 <T>와 달리 타입 소거(Type Erasure)를 넘어서 런타임에도 __orig_class__로 타입 인수에 접근 가능합니다.

05 — Protocol: 구조적 서브타이핑

핵심 비유: 오디션과 혈통

두 가지 배우 선발 방식이 있습니다:
혈통 (명목적): "연극과 졸업생만 지원 가능" — 출신 학교가 중요
오디션 (구조적): "연기를 잘하면 됩니다" — 능력만 중요

ABC는 혈통(상속)으로 판단하고, Protocol은 오디션(메서드 존재 여부)으로 판단합니다.

from typing import Protocol

class Comparable(Protocol):
    def __lt__(self, other: object) -> bool: ...
    def __le__(self, other: object) -> bool: ...
    def __gt__(self, other: object) -> bool: ...
    def __ge__(self, other: object) -> bool: ...

class Student:
    def __init__(self, name: str, score: int):
        self.name = name
        self.score = score

    def __lt__(self, other: "Student") -> bool:
        return self.score < other.score
    def __le__(self, other: "Student") -> bool:
        return self.score <= other.score
    def __gt__(self, other: "Student") -> bool:
        return self.score > other.score
    def __ge__(self, other: "Student") -> bool:
        return self.score >= other.score

def top_student(students: list[Comparable]) -> Comparable:
    return max(students)

students = [Student("철수", 85), Student("영희", 92)]
best = top_student(students)
print(best.name)   # 영희

Part B: 명목적 서브타이핑 vs 구조적 서브타이핑

명목적 서브타이핑(Nominal subtyping): Java, C++ 방식. class Dog extends Animal — 명시적 상속 선언이 있어야 서브타입.
구조적 서브타이핑(Structural subtyping): TypeScript, Go, OCaml 방식. 같은 메서드를 가지면 서브타입. Python의 Protocol이 이 방식.
Python은 두 방식을 모두 지원하는 유일한 주류 언어 중 하나입니다: ABC(명목적) + Protocol(구조적).

from typing import Protocol

# ABC 방식 — "너는 반드시 내 자식이어야 한다"
class LoggerABC(ABC):
    @abstractmethod
    def log(self, message: str) -> None: ...

class FileLogger(LoggerABC):     # 상속 필수
    def log(self, message: str) -> None:
        print(f"[FILE] {message}")

    def log(self, message: str) -> None:
        print(f"[CONSOLE] {message}")

def write_log(logger: LoggerProtocol, msg: str) -> None:
    logger.log(msg)

# 둘 다 사용 가능
write_log(FileLogger(), "ABC 방식")
write_log(ConsoleLogger(), "Protocol 방식")

07 — dataclass: 자동 생성의 원리

핵심 비유: 자동 서류 작성기

회사에 입사하면 이력서, 보험서류, 사원증을 직접 작성해야 합니다. @dataclass는 자동 서류 작성기입니다 — "이름, 나이, 부서"만 알려주면 나머지 서류(init, repr, eq)를 자동으로 만들어줍니다.

초급

@dataclass — 보일러플레이트 없이

# 수동으로 작성하면
class PersonManual:
    def __init__(self, name: str, age: int):
        self.name = name
        self.age = age

    def __repr__(self):
        return f"PersonManual(name='{self.name}', age={self.age})"

    def __eq__(self, other):
        if not isinstance(other, PersonManual):
            return NotImplemented
        return self.name == other.name and self.age == other.age

from dataclasses import dataclass

print(p1 == p2)      # True  ← __eq__ 자동 생성
print(p1.name)       # 철수

@dataclass(frozen=True)

    y: float

frozen이 아닌 일반 dataclass는 hash 가능한가요?
기본적으로 @dataclass는 __eq__를 생성하면서 __hash__ = None으로 설정합니다 (unhashable). 12주차에서 배운 것처럼 __eq__를 정의하면 기본 __hash__가 무효화됩니다. frozen=True이면 불변이 보장되므로 자동으로 __hash__가 생성됩니다. 또는 @dataclass(unsafe_hash=True)로 강제할 수 있지만 가변 객체를 해시하면 위험합니다.

# 데코레이터 타입 힌트 — ParamSpec (Python 3.10+)
from typing import ParamSpec, TypeVar
P = ParamSpec('P')
R = TypeVar('R')

def log_call(func: Callable[P, R]) -> Callable[P, R]:
    """ParamSpec으로 매개변수 시그니처를 완벽히 보존"""
    from functools import wraps
    @wraps(func)
    def wrapper(*args: P.args, **kwargs: P.kwargs) -> R:
        print(f"호출: {func.__name__}")
        return func(*args, **kwargs)
    return wrapper

@log_call
def add(a: int, b: int) -> int:
    return a + b

# mypy가 add의 시그니처를 (int, int) -> int로 정확히 인식!
# ParamSpec 없이 Callable[..., Any]를 쓰면 시그니처 정보가 사라짐
ParamSpec vs TypeVar: TypeVar는 단일 타입을 캡처합니다. ParamSpec은 함수의 전체 매개변수 목록(이름, 타입, 기본값)을 캡처합니다. 데코레이터가 원본 함수의 시그니처를 보존하려면 ParamSpec이 필수입니다.

Union[int, str]은 타입 이론의 Sum Type(합 타입)입니다. Haskell의 Either a b, Rust의 enum에 해당합니다. @dataclass는 Product Type(곱 타입)입니다. 이 둘을 합치면 Algebraic Data Type(ADT)이 됩니다.

Python 3.10의 match/case문은 ADT 패턴 매칭을 지원합니다: 

from dataclasses import dataclass

@dataclass
class Circle:
    radius: float

@dataclass
class Rectangle:
    width: float
    height: float

Shape = Circle | Rectangle   # Sum Type

def area(shape: Shape) -> float:
    match shape:              # 패턴 매칭
        case Circle(r):
            return 3.14159 * r ** 2
        case Rectangle(w, h):
            return w * h

print(area(Circle(5)))         # 78.53975
print(area(Rectangle(3, 4)))   # 12
