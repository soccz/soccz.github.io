01 — 이번 주 이야기

카페에서 주문을 받는 직원이 있습니다. 손님이 커피를 주문하면, 커피가 나올 때까지 가만히 서서 기다릴까요?

당연히 아닙니다. 커피 머신에 주문을 넣고, 다음 손님의 주문을 받습니다. 커피가 완성되면 그때 가져다줍니다.

이것이 비동기(Asynchronous) 프로그래밍입니다. 직원 한 명(단일 스레드)이 여러 주문을 동시에 처리합니다. 

# 동기 (Synchronous) — 한 번에 하나씩
result1 = fetch_from_api("서버A")   # 2초 기다림
result2 = fetch_from_api("서버B")   # 2초 기다림
# 총 4초

# 비동기 (Asynchronous) — 기다리는 동안 다른 일
result1, result2 = await asyncio.gather(
    fetch_from_api("서버A"),   # 요청 보내고
    fetch_from_api("서버B"),   # 바로 다음 요청
)
# 총 ~2초 (동시 대기)

이번 주는 Python이 제공하는 세 가지 동시성 도구 — threading, multiprocessing, asyncio — 의 차이와 내부 동작을 완전히 분해합니다.

02 — 동시성 vs 병렬성: 개념 정리

핵심 비유: 요리사

동시성(Concurrency): 요리사 1명이 파스타 삶는 동안 소스를 만든다. 한 번에 하나만 하지만, 대기 시간을 활용해 여러 일을 진행시킨다.
병렬성(Parallelism): 요리사 3명이 각자 다른 요리를 동시에 만든다. 실제로 같은 순간에 여러 일이 일어난다.

초급

먼저, 문제를 느껴봅시다

세 웹사이트에서 데이터를 가져와야 합니다:

import time

def fetch(url):
    print(f"{url} 요청 중...")
    time.sleep(1)   # 네트워크 응답 1초 대기
    return f"{url}: 데이터"

start = time.perf_counter()
r1 = fetch("api/users")      # 1초 기다림
r2 = fetch("api/posts")      # 또 1초 기다림
r3 = fetch("api/comments")   # 또 1초 기다림
elapsed = time.perf_counter() - start

print(f"총 시간: {elapsed:.1f}초")   # 3.0초!
# 각 요청은 독립적인데 왜 3초나 걸리는가?
# 두 번째 요청은 첫 번째가 끝날 때까지 아무것도 안 하고 기다림!

1초짜리 작업 3개인데 3초 걸립니다. 기다리는 동안 다음 요청을 보내면 ~1초로 줄일 수 있지 않을까요? 이것이 동시성의 핵심입니다.

동시성 ≠ 병렬성

핵심 구분:
  • 동시성: 여러 작업이 번갈아가며 실행 (1 CPU도 가능)
  • 병렬성: 여러 작업이 동시에 실행 (여러 CPU 필요)
동시성은 "구조", 병렬성은 "실행 방식"입니다.

Python의 세 가지 도구

# 1. threading — 동시성 (GIL 때문에 병렬 아님)
#    I/O 대기(네트워크, 파일)를 줄이는 데 효과적

# 2. multiprocessing — 진짜 병렬
#    CPU 연산(이미지 처리, 수학 계산)에 효과적

# 3. asyncio — 동시성 (단일 스레드)
#    대량의 I/O 대기(수천 API 호출)에 가장 효과적
중급

I/O-bound vs CPU-bound

# I/O-bound: 대부분의 시간을 기다리는 데 소비
# - 웹 API 호출, 파일 읽기/쓰기, DB 쿼리
# → threading 또는 asyncio가 효과적

# CPU-bound: 대부분의 시간을 계산에 소비
# - 이미지 처리, 암호화, 수학 연산, ML 학습
# → multiprocessing이 효과적 (또는 C 확장)
가장 흔한 실수: CPU-bound 작업에 threading을 쓰면 GIL 때문에 오히려 느려집니다. 스레드 전환 오버헤드만 추가됩니다.
전공자

Part B: 동시성 모델 — OS 이론

동시성의 역사: 프로세스(1960s, Multics) → 스레드(1980s, Mach) → 이벤트 루프(1990s, Tcl/Tk) → 코루틴(2000s, Lua/Python).

선점형(Preemptive): OS가 스레드를 강제로 전환 (threading).
협력형(Cooperative): 코드가 자발적으로 양보 (asyncio의 await).

Rob Pike(Go 언어 창시자): "Concurrency is not parallelism." 동시성은 독립적 실행 단위를 구성하는 방법이고, 병렬성은 실제로 동시에 실행하는 것입니다.

03 — GIL: 왜 존재하는가

핵심 비유: 화장실 열쇠

사무실에 화장실(메모리)이 1개 있고, 열쇠(GIL)도 1개입니다. 직원(스레드) 3명이 있지만, 열쇠를 가진 사람만 화장실을 쓸 수 있습니다. 다른 직원은 열쇠를 받을 때까지 기다려야 합니다.

이것이 Global Interpreter Lock(GIL)입니다.

중급

GIL이란 무엇인가

# GIL = Global Interpreter Lock
# Python 바이트코드를 한 번에 하나의 스레드만 실행하게 하는 뮤텍스

# CPU-bound에서 threading이 느린 이유
import threading
import time

def count_up(n):
    total = 0
    for _ in range(n):
        total += 1
    return total

N = 50_000_000

# 단일 스레드
start = time.perf_counter()
count_up(N)
single = time.perf_counter() - start

# 2개 스레드 (GIL 때문에 이득 없음)
start = time.perf_counter()
t1 = threading.Thread(target=count_up, args=(N//2,))
t2 = threading.Thread(target=count_up, args=(N//2,))
t1.start(); t2.start()
t1.join(); t2.join()
threaded = time.perf_counter() - start

print(f"단일 스레드: {single:.2f}초")    # 0.97초 (실측)
print(f"2개 스레드:  {threaded:.2f}초")  # 0.89초 (실측)
# 2개 스레드가 8% 빠를 뿐! GIL 때문에 병렬 효과 거의 없음
# (스레드 전환 오버헤드 때문에 오히려 느려질 수도 있음)
GIL의 핵심: Python 스레드가 여러 개여도, Python 바이트코드는 한 번에 하나만 실행됩니다. 단, I/O 대기 중에는 GIL을 풀어서 다른 스레드가 실행될 수 있습니다. 이것이 I/O-bound에서만 threading이 효과적인 이유입니다.
GIL을 왜 없애지 않나요?
  1. 안전성: GIL 덕분에 CPython의 참조 카운팅(Reference Counting)이 스레드 안전합니다. GIL 없이는 모든 객체에 잠금(lock)이 필요해 성능이 크게 떨어집니다.
  2. C 확장 호환성: NumPy 등 C 확장이 GIL 존재를 가정하고 작성되었습니다.
  3. 진행 중: PEP 703(Python 3.13+)에서 GIL 선택적 비활성화가 도입되고 있습니다!
고급

GIL이 풀리는 순간

# GIL이 풀리는 대표적 경우:
# 1. I/O 작업 (파일, 네트워크, sleep)
# 2. C 확장에서 Py_BEGIN_ALLOW_THREADS 호출 시
# 3. NumPy 같은 라이브러리의 행렬 연산 시

# I/O-bound에서는 threading이 효과적!
import threading
import time

def io_task(name, duration):
    time.sleep(duration)   # sleep 동안 GIL 해제!
    return f"{name} 완료"

# 순차 실행: 3초
start = time.perf_counter()
io_task("A", 1); io_task("B", 1); io_task("C", 1)
seq = time.perf_counter() - start

# 스레드 실행: ~1초
start = time.perf_counter()
threads = [
    threading.Thread(target=io_task, args=("A", 1)),
    threading.Thread(target=io_task, args=("B", 1)),
    threading.Thread(target=io_task, args=("C", 1)),
]
for t in threads: t.start()
for t in threads: t.join()
par = time.perf_counter() - start

print(f"순차: {seq:.1f}초, 스레드: {par:.1f}초")
# 순차: 3.0초, 스레드: 1.0초  ← 3배 빠름!
전공자

Part A: GIL의 CPython 구현

Part A: CPython의 GIL은 _PyRuntimeState에 저장된 뮤텍스입니다. Python 3.2+에서 GIL은 5밀리초마다 전환 기회를 줍니다 (sys.getswitchinterval()으로 확인, 기본 0.005초). 이전 Python 2에서는 100 바이트코드 명령어마다 전환했습니다. 
import sys
print(sys.getswitchinterval())   # 0.005 (5ms)
GIL 전환 시 현재 스레드는 drop_gil()을 호출하고 대기 중인 스레드가 take_gil()로 GIL을 획득합니다. ceval_gil 구조체에서 조건 변수(condition variable)로 관리됩니다.

Part B: 왜 GIL 제거에 30년이 걸리고 있는가

GIL 제거는 1990년대부터 요구되었지만 번번이 실패했습니다. 왜?

1999년 — Greg Stein의 시도: GIL을 제거하고 모든 객체에 개별 뮤텍스를 추가했습니다. 결과: 단일 스레드 성능이 40% 하락. 대부분의 Python 코드는 단일 스레드이므로 "모두가 느려지는" 해결책은 받아들여지지 않았습니다.

핵심 딜레마: GIL 덕분에 CPython의 참조 카운팅(ob_refcnt++, ob_refcnt--)은 잠금 없이 수행됩니다. GIL을 제거하면 매 참조 카운트 변경마다 원자적 연산이 필요하고, 이것은 모든 Python 코드를 느리게 합니다.

2023년 — Sam Gross의 PEP 703: 30년 만에 실용적 해결책이 나왔습니다. Biased Reference Counting: 객체를 생성한 스레드는 잠금 없이 카운트를 수정하고, 다른 스레드에서 접근할 때만 원자적 연산을 사용합니다. 대부분의 객체는 생성한 스레드에서만 사용되므로 오버헤드가 최소화됩니다.

Python 3.13(2024)에서 실험적 빌드(--disable-gil)가 도입되었고, 향후 버전에서 점진적으로 기본화될 예정이지만 정확한 일정은 미정입니다. 30년의 교훈: "GIL은 나쁘다"는 단순한 문제가 아니라, "단일 스레드 성능 vs 멀티 스레드 성능"의 트레이드오프입니다.

04 — threading: I/O 대기를 줄이는 방법

초급

스레드의 기본 사용법

import threading
import time

def download(filename):
    print(f"{filename} 다운로드 시작...")
    time.sleep(1)   # 다운로드 시뮬레이션
    print(f"{filename} 다운로드 완료!")

# 스레드 3개 생성
threads = [
    threading.Thread(target=download, args=("파일A",)),
    threading.Thread(target=download, args=("파일B",)),
    threading.Thread(target=download, args=("파일C",)),
]

start = time.perf_counter()
for t in threads:
    t.start()       # 각 스레드 시작

for t in threads:
    t.join()        # 모든 스레드 완료 대기

elapsed = time.perf_counter() - start
print(f"총 시간: {elapsed:.1f}초")   # ~1초 (순차면 3초)
start()는 스레드를 시작하고 즉시 반환합니다. join()은 해당 스레드가 끝날 때까지 기다립니다.
중급

스레드 안전성과 Lock

import threading

# 위험: 여러 스레드가 같은 변수를 수정
counter = 0

def increment():
    global counter
    for _ in range(100_000):
        counter += 1   # 읽기-수정-쓰기가 원자적이지 않음!

threads = [threading.Thread(target=increment) for _ in range(5)]
for t in threads: t.start()
for t in threads: t.join()
print(f"기대값: 500000, 실제값: {counter}")
# 500000이 아닌 경우가 있음! (Race condition)
Race Condition (경쟁 조건): counter += 1은 실제로 세 단계입니다: (1) counter 읽기 → (2) 1 더하기 → (3) counter에 쓰기. 두 스레드가 동시에 (1)을 실행하면 같은 값을 읽고, 결과가 하나만 반영됩니다. 
# 해결: Lock으로 보호
counter = 0
lock = threading.Lock()

def safe_increment():
    global counter
    for _ in range(100_000):
        with lock:          # Lock 획득 → 해제 (context manager)
            counter += 1    # 한 번에 하나의 스레드만 실행

threads = [threading.Thread(target=safe_increment) for _ in range(5)]
for t in threads: t.start()
for t in threads: t.join()
print(f"기대값: 500000, 실제값: {counter}")   # 항상 500000!
Lock 외에 다른 동기화 도구는?
  • threading.RLock: 같은 스레드가 여러 번 acquire 가능 (Reentrant Lock)
  • threading.Semaphore(n): n개의 스레드가 동시 접근 가능
  • threading.Event: 스레드 간 신호 전달
  • threading.Barrier(n): n개 스레드가 모두 도착할 때까지 대기
  • queue.Queue: 스레드 안전한 큐 (producer-consumer 패턴)
고급

daemon 스레드와 생명주기

import threading
import time

def background_task():
    while True:
        print("백그라운드 작업 중...")
        time.sleep(1)

# daemon=True: 메인 스레드 종료 시 함께 종료
t = threading.Thread(target=background_task, daemon=True)
t.start()

time.sleep(2.5)
print("메인 종료 — daemon도 종료됩니다")
# daemon 스레드는 메인이 끝나면 강제 종료됨 (정리 코드 실행 안 됨!)
daemon 주의: daemon 스레드는 finally 블록이나 context manager의 __exit__이 실행되지 않을 수 있습니다. 파일 쓰기나 DB 트랜잭션에 사용하면 위험합니다.
전공자

Part A: Python 스레드의 CPython 구현

Part A: CPython의 threading.Thread는 OS 네이티브 스레드를 래핑합니다. Linux에서는 pthread_create, Windows에서는 _beginthreadex를 호출합니다. 각 스레드는 자신의 PyThreadState를 가지지만, 바이트코드 실행은 GIL을 획득한 스레드만 가능합니다. LOAD_GLOBAL, STORE_FAST 같은 단일 바이트코드는 GIL 덕분에 원자적이지만, counter += 1은 LOAD_GLOBAL 0 (counter) → LOAD_CONST 1 (1) → BINARY_OP 13 (+=) → STORE_GLOBAL 0 (counter) 4개 명령어이므로 중간에 GIL이 전환될 수 있습니다. (Python 3.11 이하에서는 BINARY_OP 대신 BINARY_ADD였습니다.)

05 — multiprocessing: 진짜 병렬

핵심 비유: 분점 (Branch Office)

threading이 한 사무실에서 여러 직원이 일하는 것이라면, multiprocessing은 각 직원이 독립된 사무실(프로세스)을 가지는 것입니다. 각 사무실에 자체 메모리, 자체 GIL이 있어 진짜 동시에 일할 수 있습니다. 단, 사무실 간 서류를 주고받으려면(프로세스 간 통신) 비용이 듭니다.

중급

multiprocessing 기본 사용법

from multiprocessing import Pool
import time

def heavy_computation(n):
    """CPU-bound 작업"""
    total = sum(i * i for i in range(n))
    return total

numbers = [10_000_000] * 4

# 순차 실행
start = time.perf_counter()
results_seq = [heavy_computation(n) for n in numbers]
seq_time = time.perf_counter() - start

# 병렬 실행 (4 프로세스)
start = time.perf_counter()
with Pool(4) as pool:

06 — asyncio 기초: 단일 스레드인데 왜 빠른가

핵심 비유: 카페 직원 (Event Loop)

직원 1명(이벤트 루프)이 주문을 받고, 커피 머신에 넣고, 다음 손님을 받습니다. 커피가 완성되면("이벤트 발생") 그때 가져다줍니다.

직원이 여러 명이 아닙니다. 한 명이 대기 시간을 효율적으로 관리합니다. 이것이 asyncio의 이벤트 루프입니다.

초급

async와 await — 기본 문법

import asyncio

# async def = "이 함수는 비동기 함수(코루틴)입니다"
async def greet(name):
    print(f"{name}에게 인사 시작...")
    await asyncio.sleep(1)   # 1초 대기 (이 동안 다른 일 가능!)
    print(f"안녕, {name}!")
    return f"{name} 완료"

# asyncio.run()으로 실행
result = asyncio.run(greet("철수"))
print(result)   # 철수 완료
async def는 코루틴(coroutine)을 정의합니다. await는 "여기서 대기할 테니 다른 일을 해도 됩니다"라는 의미입니다. asyncio.run()이 이벤트 루프를 생성하고 코루틴을 실행합니다.
주의: 코루틴을 직접 호출하면 실행되지 않습니다! 
result = greet("철수")   # 코루틴 객체만 생성됨, 실행 안 됨!
print(type(result))      # <class 'coroutine'>

# 반드시 await 하거나 asyncio.run()으로 실행해야 합니다

동시 실행의 핵심: await가 양보하는 순간

        task("B", 1),
        task("C", 3),

Part A: 코루틴의 내부 — 제너레이터에서 코루틴으로

Part A: Python의 코루틴은 역사적으로 제너레이터에서 발전했습니다:
  1. PEP 255 (Python 2.2): yield 도입 — 제너레이터
  2. PEP 342 (Python 2.5): send(), throw() 추가 — 양방향 제너레이터
  3. PEP 380 (Python 3.3): yield from — 서브제너레이터 위임
  4. PEP 492 (Python 3.5): async def / await — 네이티브 코루틴
  5. PEP 525 (Python 3.6): async def + yield — 비동기 제너레이터
네이티브 코루틴(async def)의 내부 상태 머신은 제너레이터와 동일합니다: gi_frame(cr_frame)에 실행 위치를 저장하고, send(None)으로 재개합니다. 
async def example():
    await asyncio.sleep(0)
    return 42

print(dir(coro))           # cr_frame, cr_code, cr_origin, send, throw, close...

Part B: I/O 멀티플렉싱 — OS 레벨

asyncio의 성능은 OS의 I/O 멀티플렉싱 시스템 콜에 의존합니다:

select(2): 가장 오래된 방식. fd_set 비트마스크로 관리. O(n) 스캔. FD 1024개 제한.
poll(2): select 개선. FD 수 제한 없지만 여전히 O(n).
epoll(7) (Linux): 이벤트 기반. O(1) 알림. 수만 연결 가능. asyncio의 기본.
kqueue (macOS/BSD): epoll과 유사. 파일/소켓/프로세스 등 다양한 이벤트 통합.

이것이 asyncio가 수천~수만 개의 동시 연결을 단일 스레드로 처리할 수 있는 이유입니다. C10K 문제(동시 1만 연결)의 해결책입니다.

async def long_running():
    try:
        await asyncio.sleep(100)
    except asyncio.CancelledError:
        print("취소됨! 정리 작업 수행...")
        # DB 연결 해제, 파일 닫기 등
        raise   # CancelledError를 다시 발생시켜야 취소가 완료됨

    task = asyncio.create_task(long_running())
    await asyncio.sleep(1)

    task.cancel()         # 취소 요청
    try:
        await task        # CancelledError 발생
    except asyncio.CancelledError:
        print("태스크가 정상적으로 취소되었습니다")

asyncio.Queue — Producer-Consumer 패턴

async def producer(queue, n):
    for i in range(n):
        await asyncio.sleep(0.1)
        await queue.put(f"item-{i}")
        print(f"생산: item-{i}")
    await queue.put(None)   # 종료 신호

async def consumer(queue):
    while True:
        item = await queue.get()
        if item is None:
            break
        print(f"소비: {item}")
        queue.task_done()

async def main():
    queue = asyncio.Queue(maxsize=5)   # 최대 5개 버퍼
    await asyncio.gather(
        producer(queue, 10),
        consumer(queue),
    )

asyncio.run(main())

import asyncio

class AsyncDB:

        await asyncio.sleep(0.1)   # 연결 대기

        print("DB 연결 해제 완료!")
        return False

    async def query(self, sql):
        await asyncio.sleep(0.1)
        return f"결과: {sql}"

        print(result)
    # __aexit__ 자동 호출

# DB 연결 완료!

# DB 연결 해제 중...
# DB 연결 해제 완료!

from concurrent.futures import ThreadPoolExecutor, ProcessPoolExecutor
import time

def download(url):

    return f"{url}: 완료"

with ThreadPoolExecutor(max_workers=5) as executor:

    return sum(i*i for i in range(n))

    futures = [executor.submit(compute, 5_000_000) for _ in range(4)]
    results = [f.result() for f in futures]

Future 객체와 as_completed

from concurrent.futures import ThreadPoolExecutor, as_completed
import time

def fetch(name, delay):
    time.sleep(delay)
    return f"{name}: {delay}초"

with ThreadPoolExecutor(max_workers=3) as executor:
    futures = {
        executor.submit(fetch, "A", 3): "A",
        executor.submit(fetch, "B", 1): "B",
        executor.submit(fetch, "C", 2): "C",
    }

    # as_completed: 완료된 순서대로 처리
    for future in as_completed(futures):
        name = futures[future]
        print(f"{name} 완료: {future.result()}")
    # B 완료: B: 1초
    # C 완료: C: 2초
    # A 완료: A: 3초
concurrent.futures는 threading과 multiprocessing의 통합 인터페이스입니다. Executor만 교체하면 스레드↔프로세스를 쉽게 전환합니다.
max_workers 선택 기준:
  • ThreadPoolExecutor (I/O-bound): 기본값 min(32, os.cpu_count() + 4) (Python 3.13+: min(32, (os.process_cpu_count() or 1) + 4)). I/O 대기가 길면 코어 수 × 5~10까지 올릴 수 있음. 너무 많으면 컨텍스트 스위칭 오버헤드
  • ProcessPoolExecutor (CPU-bound): 기본값 os.cpu_count() (Python 3.13+: os.process_cpu_count()). 코어 수를 초과하면 프로세스 전환 비용만 증가

import asyncio
from concurrent.futures import ThreadPoolExecutor

def blocking_io(name):
    """동기 I/O 함수 (수정 불가한 라이브러리)"""
    import time

        loop.run_in_executor(executor, blocking_io, "A"),

        loop.run_in_executor(executor, blocking_io, "C"),

Flask/Django는 왜 asyncio를 안 쓰나요?
전통적인 웹 프레임워크는 WSGI(동기) 기반입니다. 각 요청을 별도의 스레드나 프로세스로 처리합니다. Django 3.1+에서 async 뷰를 지원하기 시작했고, FastAPI/Starlette은 처음부터 ASGI(비동기) 기반입니다. ASGI 프레임워크는 단일 프로세스로도 수천 동시 연결을 처리합니다.
고급

성능 비교: 100개 I/O 작업

import asyncio
import time
from concurrent.futures import ThreadPoolExecutor

N = 100
DELAY = 0.1

# 1. 순차 — 10초
# for _ in range(100): time.sleep(0.1)

# 2. threading — ~0.2초
def sync_task(): time.sleep(DELAY)
start = time.perf_counter()
with ThreadPoolExecutor(max_workers=100) as ex:
    list(ex.map(lambda _: sync_task(), range(N)))
threading_time = time.perf_counter() - start

# 3. asyncio — ~0.1초
async def async_task(): await asyncio.sleep(DELAY)
async def async_main():
    await asyncio.gather(*[async_task() for _ in range(N)])
start = time.perf_counter()
asyncio.run(async_main())
asyncio_time = time.perf_counter() - start

print(f"순차:      {N * DELAY:.1f}초")
print(f"threading: {threading_time:.3f}초 ({N} 스레드)")  # ~0.1초 + 스레드 생성 오버헤드
print(f"asyncio:   {asyncio_time:.3f}초 (1 스레드)")      # ~0.1초 + 이벤트루프 오버헤드
# 순차 대비 ~100배 빠름! asyncio가 스레드보다 약간 더 빠르면서 메모리는 훨씬 적음
전공자

Part A: 스레드 vs 코루틴 — 메모리와 전환 비용

Part A: OS 스레드: 스택 크기 ~8MB (Linux 기본), 컨텍스트 스위칭 ~수 마이크로초. 코루틴: 초기 크기 ~100–200바이트 (코루틴 객체 셸 기준, 프레임 포함 시 더 큼), 전환은 Python 레벨 함수 호출과 동일. 따라서 1만 스레드 = ~80GB 가상 메모리 예약 (물리 메모리는 실사용량만큼만 commit), 1만 코루틴 = ~2MB. 이것이 asyncio가 대량 동시 연결에 적합한 이유입니다. sys.getsizeof로 코루틴 크기를 확인할 수 있습니다.

Part B: 왜 구조화된 동시성(Structured Concurrency)이 필요했는가

1960년대 goto가 코드를 스파게티로 만든 것처럼, 전통적 동시성(threading, asyncio.create_task)도 같은 문제를 가집니다: 

# 문제: 태스크를 "발사"하고 추적을 잃어버림
async def problematic():
    task = asyncio.create_task(some_work())   # 발사!
    # ... 여기서 예외가 발생하면?
    # task는 어디서 실행되고 있는가?
    # task의 결과는 누가 처리하는가?
    # task에서 예외가 나면 누가 잡는가?

Dijkstra가 goto를 if/while/for로 대체한 것처럼, Martin Sústrik(2016)과 Nathaniel J. Smith(2018, Trio)가 동시성에 같은 원칙을 적용했습니다: "동시 작업은 스코프가 있어야 한다. 스코프가 끝나면 모든 자식 태스크가 끝나야 한다."

Python의 asyncio.TaskGroup(3.11+)이 이 패턴을 구현합니다. Kotlin의 coroutineScope, Java 21의 StructuredTaskScope도 같은 개념. 핵심: 부모 스코프가 끝나면 모든 자식 태스크가 완료/취소됨을 보장합니다.

11 — AI와 동시성

AI 서비스는 비동기의 대표 사례입니다: LLM API 호출은 수초가 걸리는 I/O-bound 작업입니다. 여러 프롬프트를 동시에 보내면 처리량이 극적으로 향상됩니다.

LLM API 동시 호출

import asyncio

# AI API 호출 시뮬레이션
async def call_llm(prompt, delay=2):
    """LLM API 호출 (2초 소요)"""
    await asyncio.sleep(delay)
    return f"AI 응답: {prompt[:20]}..."

async def main():
    prompts = [
        "파이썬에서 리스트란 무엇인가?",
        "for 문은 어떻게 동작하는가?",
        "클로저와 데코레이터의 차이는?",
        "GIL이란 무엇인가?",
        "asyncio는 왜 빠른가?",
    ]

    # 순차: 5 × 2초 = 10초
    # 동시: ~2초!
    results = await asyncio.gather(
        *[call_llm(p) for p in prompts]
    )

    for prompt, result in zip(prompts, results):
        print(f"Q: {prompt[:15]}... → {result}")

asyncio.run(main())

# FastAPI + AI — 실제 패턴
from fastapi import FastAPI
import asyncio

app = FastAPI()

async def generate_response(prompt: str) -> str:
    # 실제로는 OpenAI/Anthropic API 호출
    await asyncio.sleep(1)   # API 응답 대기
    return f"AI 답변: {prompt}"

@app.post("/chat")
async def chat(prompt: str):
    # 비동기이므로 다른 요청도 동시 처리 가능!
    response = await generate_response(prompt)
    return {"response": response}

# 동시에 1000명이 /chat을 호출해도
# 단일 프로세스에서 처리 가능 (asyncio 덕분)
AI 시대의 동시성: 바이브코딩에서 AI에게 "API 100개를 동시 호출해줘"라고 하면 AI가 asyncio 코드를 생성합니다. 하지만 time.sleep과 asyncio.sleep을 혼용하거나, await를 빠뜨리는 실수를 할 수 있습니다. 동시성의 원리를 이해해야 AI가 생성한 비동기 코드의 정확성을 검증할 수 있습니다.

실습: 비동기 웹 크롤러 뼈대

import asyncio

async def fetch_page(url, delay=0.5):
    """웹 페이지 다운로드 시뮬레이션"""
    await asyncio.sleep(delay)
    return f"<html>{url} 내용</html>"

async def parse(html):
    """HTML 파싱 시뮬레이션"""
    await asyncio.sleep(0.1)
    return html.replace("<html>", "").replace("</html>", "")

async def crawl(urls):
    """여러 URL을 동시에 크롤링"""
    # 1단계: 동시 다운로드
    pages = await asyncio.gather(
        *[fetch_page(url) for url in urls]
    )

    # 2단계: 동시 파싱
    results = await asyncio.gather(
        *[parse(page) for page in pages]
    )

    return results

urls = [f"https://example.com/page/{i}" for i in range(10)]
results = asyncio.run(crawl(urls))
for r in results[:3]:
    print(r)

문제 10 (고급): Producer-Consumer with asyncio.Queue 
import asyncio

async def producer(queue, items):
    for item in items:
        await asyncio.sleep(0.1)
        await queue.put(item)
        print(f"생산: {item}")
    await queue.put(None)

async def consumer(queue, name):
    while True:
        item = await queue.get()
        if item is None:
            await queue.put(None)   # 다른 consumer에 전파
            break
        await asyncio.sleep(0.2)
        print(f"[{name}] 소비: {item}")

async def main():
    queue = asyncio.Queue(maxsize=3)
    await asyncio.gather(
        producer(queue, list(range(8))),
        consumer(queue, "C1"),
        consumer(queue, "C2"),
    )

asyncio.run(main())

13 — 요약 & 용어집

핵심 개념 요약표

개념핵심비유적합한 상황
동시성 번갈아 실행 요리사 1명, 여러 요리 I/O-bound
병렬성 동시 실행 요리사 3명 CPU-bound
GIL 바이트코드 잠금 화장실 열쇠 1개 —
threading OS 스레드 한 사무실 여러 직원 I/O-bound (소규모)
multiprocessing 별도 프로세스 각자 사무실 (분점) CPU-bound
asyncio 이벤트 루프 카페 직원 1명 I/O-bound (대규모)
concurrent.futures 통합 인터페이스 범용 리모컨 스레드/프로세스 전환

용어집 (32개)

동시성 (Concurrency) 여러 작업이 번갈아가며 진행되는 것. 1 CPU로도 가능. 구조적 개념
병렬성 (Parallelism) 여러 작업이 물리적으로 동시에 실행되는 것. 여러 CPU 필요
I/O-bound 대부분의 시간을 I/O 대기(네트워크, 파일)에 소비하는 작업
CPU-bound 대부분의 시간을 CPU 연산(계산, 처리)에 소비하는 작업
GIL (Global Interpreter Lock) CPython에서 한 번에 하나의 스레드만 바이트코드를 실행하게 하는 뮤텍스
Thread (스레드) 프로세스 내의 실행 단위. 같은 메모리 공간 공유. OS가 스케줄링
Process (프로세스) 독립된 메모리 공간을 가진 실행 단위. 각 프로세스에 자체 GIL
코루틴 (Coroutine) async def로 정의된 비동기 함수. await로 실행을 양보할 수 있음
이벤트 루프 (Event Loop) asyncio의 핵심. 코루틴 스케줄링, I/O 이벤트 감시, 콜백 실행을 관리
await 코루틴의 실행을 일시 중지하고 이벤트 루프에 제어를 양보하는 키워드
asyncio.run() 이벤트 루프를 생성하고 코루틴을 실행하는 진입점 함수
asyncio.gather() 여러 코루틴을 동시 실행하고 모든 결과를 리스트로 반환
asyncio.create_task() 코루틴을 Task로 감싸서 이벤트 루프에 즉시 스케줄링
asyncio.TaskGroup Python 3.11+의 구조화된 동시성 도구. 모든 태스크의 완료/취소를 보장
asyncio.Queue 비동기 Producer-Consumer 패턴을 위한 코루틴 안전 큐 (스레드 안전 아님 — 스레드용은 queue.Queue)
asyncio.wait_for() 코루틴에 타임아웃을 설정. 초과 시 TimeoutError 발생
Lock (잠금) 한 번에 하나의 스레드만 임계 영역에 접근하게 하는 동기화 도구
Race Condition (경쟁 조건) 여러 스레드가 공유 자원에 동시 접근하여 결과가 예측 불가한 상황
Deadlock (교착 상태) 두 스레드가 서로의 Lock을 기다려 영원히 진행되지 않는 상태
daemon 스레드 메인 스레드 종료 시 함께 종료되는 배경 스레드. 정리 코드가 실행 안 될 수 있음
ThreadPoolExecutor concurrent.futures의 스레드 풀. I/O-bound 작업에 적합
ProcessPoolExecutor concurrent.futures의 프로세스 풀. CPU-bound 작업에 적합
Future 아직 완료되지 않은 작업의 결과를 나타내는 객체. result()로 결과 수집
run_in_executor() 동기 함수를 asyncio 이벤트 루프에서 스레드/프로세스풀로 실행
epoll / kqueue OS 레벨 I/O 멀티플렉싱. asyncio 이벤트 루프의 기반 기술
선점형 멀티태스킹 (Preemptive) OS가 스레드를 강제 전환. threading이 이 방식
협력형 멀티태스킹 (Cooperative) 코드가 자발적으로 양보(await). asyncio가 이 방식
PEP 703 GIL-free Python 제안. Python 3.13+에서 실험적 지원
구조화된 동시성 (Structured Concurrency) 동시 작업의 시작과 끝을 스코프로 보장하는 패턴. TaskGroup이 구현
async for 비동기 이터레이터를 순회하는 문법. __aiter__/__anext__ 사용
async with 비동기 컨텍스트 매니저 문법. __aenter__/__aexit__ 사용
