Starting Point

뭘 모르는지도 모르는 상태

처음 자취를 알아보면서 느낀 건, 부동산 시장의 정보 비대칭이 구조적이라는 것이었습니다. 집주인과 중개사는 수십 번 해본 거래를, 나는 태어나서 처음 합니다. 뭘 물어봐야 하는지조차 모르니까, "괜찮아 보이는데?" 하고 넘어갈 수밖에 없습니다.

"월세 50만원"이라고 적힌 매물. 그런데 내가 중소기업에 다니는 만 28세라면, 중기청 전세대출(금리 1.2%)로 전세 전환이 가능하고, 청년월세지원(월 20만원)까지 더하면 실질 부담은 완전히 다른 숫자가 됩니다. 이걸 아는 사람과 모르는 사람의 격차가 매달 수십만원입니다.

그 격차를 기술로 줄일 수 있지 않을까?
검색으로 끝나지 않는 문제
매물을 찾는 건 시작일 뿐입니다. 등기부등본은 어떻게 읽어? 근저당이 높으면 뭐가 위험해? 전세보증보험은 꼭 들어야 해? 집주인한테 뭐라고 말해야 해? 각 단계마다 전혀 다른 종류의 지식이 필요합니다.
흩어져 있는 혜택
청년 주거 지원 제도가 15개 넘게 있지만, 내 나이 · 소득 · 자산 · 직장 유형에 따라 받을 수 있는 게 다릅니다. 조건을 하나하나 대조하는 건 사람이 할 일이 아니라, 시스템이 할 일이라고 생각했습니다.
사후 대응의 부재
깡통전세 피해 뉴스를 보면, 사전에 알 수 있었던 신호가 있었던 경우가 대부분입니다. 등기부의 근저당 비율, 소유자-임대인 불일치, 경매 진행 이력. 문서를 읽을 줄 알기만 하면 피할 수 있는 위험이 많았습니다.
Questions

이걸 만들면서 공부하고 싶었던 것들

단순히 문제를 해결하는 것만큼, 어떻게 해결할 것인가에 관심이 있었습니다. 이 프로젝트는 네 가지 기술적 질문에 대한 실험이기도 합니다.

하나의 거대한 프롬프트 vs 여러 전문가
GPT에게 "집 구하기 전체를 도와줘"라고 하면 어중간한 답이 나옵니다. 대신 매물 추천 · 안전 진단 · 금융 계산 · 법률 상담 · 협상 보조를 각각 독립된 에이전트로 나누면? Multi-Agent System이 단일 LLM보다 실질적으로 나은지 직접 확인하고 싶었습니다.
RAG가 정말 검색보다 나은가
15개 정부 혜택을 단순 조건 필터로도 매칭할 수 있습니다. 그런데 "나 대학원생인데 돈이 없어"처럼 비정형 입력이 들어오면? 키워드 매칭은 한계가 있고, 의미 기반 벡터 검색이 필요해집니다. ChromaDB + 한국어 임베딩(ko-sbert)의 실제 성능을 테스트했습니다.
LLM 비용을 어디까지 줄일 수 있는가
학생 프로젝트에서 API 비용은 현실적인 제약입니다. 모든 에이전트를 gpt-4o-mini(temperature 0)로 통일하고, 임베딩은 무료 로컬 모델, 금융 계산은 API 없이 로컬에서, 자가진단은 순수 연산으로. "LLM이 꼭 필요한 곳에만 LLM을 쓴다"는 원칙을 세웠습니다.
"긴 여정"을 하나의 앱으로 담을 수 있는가
집 구하기는 한 번의 검색이 아니라, 몇 주에 걸친 여정입니다. 온보딩에서 프로필을 수집하고, 이후 9개 페이지가 그 맥락을 공유하며 동작하는 구조. session_state 기반 상태 관리로 페이지 간 데이터를 이어붙이는 설계를 고민했습니다.
Design Decisions

그래서 이렇게 설계했습니다

5명의 에이전트가 각자의 전문 분야를 담당하고, LangGraph의 StateGraph 위에서 상태를 넘기며 추론합니다. 하나의 에이전트 안에서도 4단계 노드(수집 → 분석 → 검증 → 생성)가 순서대로 실행됩니다.

Agent 1 — Recommender
매물 + 혜택 추천
나이, 소득, 자산, 직장 유형을 받아서 15개 정부 혜택 중 자격이 되는 것을 RAG로 매칭합니다. 동시에 35개 이상의 매물에서 예산 · 통근 · 위험도 기준으로 필터링하고, 서울 15개 구별 지역 가이드도 함께 보여줍니다.
RAGChromaDBko-sbert지역 가이드
Agent 2 — Safety Analyzer
계약 안전 진단
등기부등본 PDF를 업로드하면 4단계로 분석합니다. 데이터 추출 → 위험도 스코어링(근저당, 압류, 가처분) → 등기부-계약서 교차 검증(소유자 불일치 = 사기 징후) → 마크다운 리포트 생성. API 없이 쓸 수 있는 자가진단 탭과 SOS 긴급대응 탭도 만들었습니다.
PDF 파싱깡통전세 탐지교차 검증SOS
Agent 3 — Negotiator
협상 메시지 생성
"집주인한테 뭐라고 말하지?"를 해결합니다. 전세보증보험 가입 요청, 특약 추가, 수리 요청, 조건 변경, 보증금 반환 내용증명까지 5종. 법적 근거가 포함된 정중하면서 단호한 톤으로, 복사해서 바로 보낼 수 있는 메시지를 만듭니다.

왜 5개로 나눴는가

하나의 프롬프트에 모든 지식을 넣으면, 맥락이 길어질수록 답변 품질이 떨어졌습니다. 에이전트별로 전문 분야를 한정하고, 시스템 프롬프트에 해당 도메인의 법률 · 데이터 · 판단 기준만 넣으니 정확도가 올라갔습니다. 트레이드오프는 에이전트 간 상태 전달의 복잡성인데, LangGraph의 StateGraph가 이 부분을 깔끔하게 처리해줬습니다.

What I Built

온보딩부터 입주까지, 9개 페이지

처음 앱을 열면 4단계 온보딩 위저드가 사용자의 상황(신분, 소득, 자산, 주요 고민)을 파악하고, 그 사람에게 맞는 커스텀 로드맵을 자동 생성합니다. "사기가 걱정돼요"를 선택하면 안전 진단 중심으로, "예산이 부족해요"를 선택하면 금융 시뮬레이션 중심으로. 이후 모든 페이지가 이 프로필을 기반으로 동작합니다.

What I Learned

만들면서 알게 된 것들

해커톤에서 첫 버전을 만들고, 이후 계속 고도화하면서 배운 것들입니다. 기술적 선택 뒤에는 항상 트레이드오프가 있었습니다.

에이전트 분리의 효과

하나의 프롬프트에 모든 것을 넣으면 "대략적으로 맞는 답"이 나옵니다. 에이전트를 분리하니 각 도메인에서 "정확하게 맞는 답"이 나왔습니다. 특히 안전 진단에서 교차 검증(등기부 소유자 vs 계약서 임대인) 같은 구체적 로직은 전담 에이전트가 아니면 빠뜨리기 쉽습니다.

RAG vs 규칙 기반의 경계

15개 혜택 매칭에서, 조건이 명확한 것(나이 19-34세, 소득 중위 60% 이하)은 규칙 기반 필터가 더 정확했습니다. 하지만 "돈이 없는 대학원생"처럼 비정형 입력이 들어오면 의미 기반 검색이 필요했습니다. 결국 RAG를 기본으로 하되, 키워드 기반 폴백을 두는 하이브리드 구조가 가장 안정적이었습니다.

LLM을 안 쓰는 것도 설계다

금융 시뮬레이터는 API를 한 번도 호출하지 않습니다. 대출 금리 계산, DSR 진단, 생활비 시뮬레이션은 수학 공식이지 언어 문제가 아닙니다. 깡통전세 자가진단도 근저당/시세 비율 계산이니까 순수 연산으로 충분합니다. "LLM이 필요한 곳"과 "규칙으로 되는 곳"을 구분하는 판단이 비용과 정확도 모두에 영향을 줬습니다.

상태 관리가 UX를 결정한다

9개 페이지가 하나의 여정으로 느껴지려면, 온보딩에서 입력한 프로필이 모든 곳에서 살아있어야 합니다. Streamlit의 session_state로 페이지 간 데이터를 이어붙이고, 사이드바에서 프로필을 수정하면 즉시 반영되는 구조를 만들었습니다. 기술적으로 복잡하진 않지만, 이 흐름이 없으면 "9개 분리된 도구"가 됩니다.
